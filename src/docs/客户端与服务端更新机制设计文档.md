# 客户端与服务端更新机制设计文档

本文描述 BiSheng **Latte（Windows 客户端）** 与 **Server（自托管服务端）** 的版本发布与升级方案，对齐现有：

- [部署文档](./部署文档.md) §3 发布、`/opt/bisheng`、§9.2 更新
- [服务端备份与恢复运维指南](./服务端备份与恢复运维指南.md) / [`scripts/server-backup/`](../../scripts/server-backup/)
- [`scripts/release/`](../../scripts/release/) / [`scripts/server-upgrade/`](../../scripts/server-upgrade/)
- README 中 Latte `dotnet publish … -r win-x64`

**目标读者：** 维护个人自建实例的运维者与后续增强自动更新的开发者。

**非目标：** 多租户 SaaS 自动滚动发布；服务端进程内静默从 GitHub 覆盖自身。

**文档状态：** P1 主路径已落地（统一发版、Velopack、升级脚本）；下文「可选 / P2+」为后续增强。

---

## 一、结论摘要

| 组件 | 推荐机制 | 用户体验 | 现状 |
|------|----------|----------|------|
| **Latte** | **Velopack** + GitHub Releases | 「关于」中检查更新 → 确认 → 下载安装 → 重启 | 已接入；数据在 `%LocalAppData%\BiSheng\Latte\` |
| **Server** | **`upgrade-bisheng.sh`**（备份 → 停服 → 校验哈希 → 替换 → 启服） | SSH/本地执行；管理页「检查更新」仍为可选 | 脚本与部署文档 §9.2 已就绪 |

原则：

1. 安装包/压缩包校验 **SHA-256**（后续可加 Authenticode / 发布者签名）
2. 客户端**不静默升级**（个人笔记需用户确认）
3. 服务端**不**由 Kestrel 进程自下载自替换
4. Latte 与 Server **共用** `BiShengVersion`、同一 GitHub Release；破坏性协议变更仍可用 `minClient` / `minServer`（清单见 §2.3，尚未强制）

## 二、版本与发布通道

### 2.1 版本号

采用 **SemVer**（`MAJOR.MINOR.PATCH`），与程序集信息一致：

| 产物 | 版本来源 |
|------|----------|
| Latte / Server | 根目录 `Directory.Build.props` 的 **`BiShengVersion`**（两端 csproj 引用同一值） |
| 共享协议 | `BiSheng.Shared` 可单独标注；破坏协议时抬 Server `minClient` |

Git tag：**单一** `v1.2.3`，一个 GitHub Release 同时挂 Latte（Velopack）与 Server（zip）。

### 2.2 发布通道：GitHub Releases

仓库：`https://github.com/hillsburg/BiSheng`。

每个 Release 典型资产（以 `scripts/release/pack-all.ps1` / Velopack 实际产出为准）：

```text
BiSheng.Latte-*-Setup.exe / *.nupkg / releases.*.json / RELEASES   # Velopack
BiSheng.Server-{version}-linux-x64.zip                            # framework-dependent
SHA256SUMS.txt                                                    # 各文件 sha256
update-manifest.json                                              # 可选，见 §2.3（尚未由 CI 自动生成）
```

> Server 使用 **framework-dependent**（`--self-contained false`），目标机需已装 .NET 8 ASP.NET Core Runtime。更新包保持同一模式。

Release 正文：CI 使用 `gh release create --generate-notes`，按上一 tag 以来的 PR/commit 自动生成；可事后在 GitHub 上 Edit。

### 2.3 清单文件 `update-manifest.json`（可选，P2）

尚未由发版流水线强制产出。规划放在 Release 资产中，并可用「最新稳定」固定 URL（例如最新 Release 资产，或 Pages 上的 `latest-update-manifest.json`）。

```json
{
  "schemaVersion": 1,
  "channel": "stable",
  "publishedAt": "2026-07-19T00:00:00Z",
  "latte": {
    "version": "1.2.3",
    "minOs": "Windows 10",
    "feedUrl": "https://github.com/hillsburg/BiSheng/releases/download/v1.2.3/",
    "setupFile": "BiSheng.Latte-win-Setup.exe",
    "sha256": "…",
    "releaseNotesUrl": "https://github.com/hillsburg/BiSheng/releases/tag/v1.2.3",
    "minServer": "1.0.0"
  },
  "server": {
    "version": "1.2.3",
    "runtime": "net8.0",
    "rid": "linux-x64",
    "packageFile": "BiSheng.Server-1.2.3-linux-x64.zip",
    "downloadUrl": "https://github.com/hillsburg/BiSheng/releases/download/v1.2.3/BiSheng.Server-1.2.3-linux-x64.zip",
    "sha256": "…",
    "minClient": "1.0.0",
    "releaseNotesUrl": "https://github.com/hillsburg/BiSheng/releases/tag/v1.2.3"
  }
}
```

- `minServer`：此 Latte 要求的最低服务端版本（可选，用于连接时提示）
- `minClient`：此 Server 要求的最低客户端版本（可选）

首期客户端更新依赖 Velopack 直连 GitHub Releases；服务端升级依赖运维下载 zip + 脚本。

## 三、服务端升级设计（对齐现有 publish / `/opt/bisheng`）

### 3.1 构建与首次部署

构建（开发机或 CI `pack-server.ps1`）：

```bash
cd src/BiSheng.Server
dotnet publish -c Release -r linux-x64 -o ./publish --self-contained false
```

首次部署：上传至 `/opt/bisheng/`，保留后续生成的 `bisheng.db`、`uploads/`、`appsettings.Production.json`、`data-protection-keys/`。

### 3.2 已实现：`upgrade-bisheng.sh`

路径：[`scripts/server-upgrade/upgrade-bisheng.sh`](../../scripts/server-upgrade/)，说明见同目录 [README](../../scripts/server-upgrade/README.md)。部署文档 §9.2 已推荐此路径。

```text
输入：
  PACKAGE=/path/to/BiSheng.Server-{ver}-linux-x64.zip
  或 DOWNLOAD_URL=… + EXPECTED_SHA256=…
  BISHENG_HOME=/opt/bisheng（默认）

步骤：
  1. 预检：systemd 单元存在；sqlite3 / unzip 可用；磁盘空间
  2. 备份：调用 backup-bisheng.sh（升级前必须成功；可用 SKIP_BACKUP=1 跳过，生产不建议）
  3. systemctl stop bisheng
  4. 校验 zip 的 sha256（若提供 EXPECTED_SHA256）
  5. 解压到临时目录；归档当前程序文件为 pre-upgrade-{stamp}.tar.gz
  6. 保留名单（不得被覆盖删除）：
       bisheng.db*、uploads/、appsettings.Production.json、
       data-protection-keys/、backup/
  7. 将 stage 内程序文件同步到 $BISHENG_HOME
  8. chown（若环境需要）
  9. systemctl start bisheng
 10. 健康检查：curl HEALTH_URL（默认 /admin/login；日后可改为 /health）
 11. 失败：尝试用 pre-upgrade-*.tar.gz 回滚程序文件；必要时 restore-bisheng.sh
```

环境变量：

| 变量 | 默认 | 说明 |
|------|------|------|
| `BISHENG_HOME` | `/opt/bisheng` | 安装根 |
| `PACKAGE` / 脚本参数 | （必填之一） | 本地 zip 路径 |
| `DOWNLOAD_URL` | （必填之一） | 远程 zip |
| `EXPECTED_SHA256` | （强烈建议） | 包哈希 |
| `SKIP_BACKUP` | `0` | 仅调试用，生产禁止 |
| `HEALTH_URL` | 含 `/admin/login` | 启服探测 |

运维示例：

```bash
# 一次性安装脚本
sudo cp scripts/server-backup/backup-bisheng.sh /usr/local/bin/
sudo cp scripts/server-upgrade/upgrade-bisheng.sh /usr/local/bin/
sudo chmod +x /usr/local/bin/backup-bisheng.sh /usr/local/bin/upgrade-bisheng.sh

# 从 Release 取得 zip 与 SHA256SUMS 后
sudo BISHENG_HOME=/opt/bisheng EXPECTED_SHA256=… \
  /usr/local/bin/upgrade-bisheng.sh /tmp/BiSheng.Server-0.1.1-linux-x64.zip
```

手工 `scp` + `cp`（不停用脚本时的备选）仍见部署文档 §9.2。

### 3.3 与发版产物的对应关系

本地或 CI：`.\scripts\release\pack-all.ps1`（或单独 `pack-server.ps1`）产出 `BiSheng.Server-{version}-linux-x64.zip` 与 `SHA256SUMS.txt`，随 tag `v*` 的 `release.yml` 上传到同一 GitHub Release。

### 3.4 回滚

| 失败点 | 动作 |
|--------|------|
| 备份失败 | 中止，不替换 |
| 哈希不符 | 中止，不替换 |
| 启服失败 / 健康检查失败 | 停服；用 `pre-upgrade-*.tar.gz` 还原程序文件；DB 一般不回滚除非迁移破坏数据 |
| 迁移破坏数据 | `restore-bisheng.sh` 指向升级前 `backup/` 目录（见备份指南） |

### 3.5 管理页「检查更新」（可选，P2）

- 端点示例：`GET /admin/api/update-check`（需管理员 Cookie）或只读配置页展示
- 服务端请求可配置的 `Update:ManifestUrl`（默认指向 `update-manifest.json`）
- **仅展示**「当前版本 / 最新版本 / 下载链接 / 发布说明」，**不**在服务器上下载执行
- 真正升级仍由运维跑 `upgrade-bisheng.sh`

## 四、Latte 客户端升级设计（Velopack）

### 4.1 现有发布

```bash
dotnet publish src/BiSheng.Latte/BiSheng.Latte.csproj -c Release -r win-x64 --self-contained false
```

目标机需 [.NET 8 Desktop Runtime](https://dotnet.microsoft.com/download/dotnet/8.0)。发版流水线在 publish 之后执行 `vpk pack`（见 `scripts/release/pack-latte.ps1`）。

> **注意：** `vpk download` / `vpk pack` 必须显式 `--outputDir` 指向同一产物目录，且勿在产物目录内依赖默认相对路径 `Releases`，以免 Windows 上与 `RELEASES` 文件名冲突。

### 4.2 为什么用 Velopack

- Squirrel.Windows 事实继任者，.NET / WPF 生态成熟
- 支持安装、差分/全量更新、重启应用
- 对接公开仓库 GitHub Releases 作为 feed（私有仓需额外 token，不宜写进客户端）

### 4.3 用户流程（已实现）

```text
「关于」→ 检查更新
  → Velopack 查询 GitHub Releases（hillsburg/BiSheng）
  → 非 Velopack 安装版：提示无法自动更新（如 dotnet run / 绿色解压）
  → 无新版本：已是最新
  → 有新版本：确认后下载 → 安装 → 退出并重启
```

约束：

- **默认不自动下载安装**（需用户确认）
- 更新前尽量：退出前 Flush 同步（已有 `FlushOnExit`）
- 更新源仅 HTTPS

### 4.4 工程接入（已落地）

1. NuGet `Velopack` + 入口 `VelopackApp.Build().Run()`（`Program.cs`）
2. `AppUpdateService` + `AboutWindow`；工具栏「关于」在连接状态徽章旁
3. CI：`release.yml` → `pack-all.ps1` → `gh release create --generate-notes`
4. 数据目录：`%LocalAppData%\BiSheng\Latte\`（`LatteAppPaths`）；首次启动可将 exe 旁旧 `local.db` / `config.json` / `images` 迁入（源改名为 `*.bak-before-appdata`）

### 4.5 与服务端版本提示（可选，P2）

Latte 连接成功后可读服务端轻量字段（例如 verify 扩展）中的 `serverVersion` + `minClient`：

- 客户端过旧 → 提示升级 Latte  
- 服务端过旧 → 提示运维升级 Server（链到部署文档 / Release）

首期可不做协议字段，仅靠 Release 说明与运维流程。

## 五、CI / 发版流水线（已落地）

[`.github/workflows/ci.yml`](../../.github/workflows/ci.yml)：PR / push 跑测试（含超时，避免 WPF Dispatcher 死锁挂死）。

[`.github/workflows/release.yml`](../../.github/workflows/release.yml)：

```text
on:
  push:
    tags: ['v*']
  workflow_dispatch:  # 输入版本号（不含 v）

jobs:
  - test（timeout-minutes: 15）
  - release：pack-all.ps1 → gh release create --generate-notes（Latte + Server + SHA256SUMS）
```

本地：`.\scripts\release\pack-all.ps1` 后可用 `gh release create` / `gh release upload` 手工上传。

## 八、风险与缓解

| 风险 | 缓解 |
|------|------|
| 升级中断半套 dll | stage + `pre-upgrade-*.tar.gz` 回滚 |
| 覆盖 `appsettings.Production.json` | 保留名单；脚本显式跳过 |
| DP keys 丢失导致 Cookie 失效 | 备份包含 `data-protection-keys/`；升级不删该目录 |
| 假更新包 | HTTPS + SHA-256；后续代码签名 |
| Velopack 覆盖用户数据 | 数据目录与安装目录分离（LocalAppData） |
| 协议不兼容 | 规划 manifest `minClient` / `minServer` + Release 说明 |
| CI Latte 测试 Dispatcher 死锁 | 导航 Publish 测试走 WpfSta；`release`/`ci` test 作业 `timeout-minutes` |
| Windows 上 `Releases` 目录与 `RELEASES` 文件冲突 | pack-latte 显式 `--outputDir`，禁止误建嵌套 `Releases\` |
