# 客户端与服务端更新机制设计文档

本文描述 BiSheng **Latte（Windows 客户端）** 与 **Server（自托管服务端）** 的版本发布与升级方案，对齐现有：

- [部署文档](./部署文档.md) §3 发布、`/opt/bisheng`、§9.2 更新
- [服务端备份与恢复运维指南](./服务端备份与恢复运维指南.md) / [`scripts/server-backup/`](../../scripts/server-backup/)
- README 中 Latte `dotnet publish … -r win-x64`

**目标读者：** 维护个人自建实例的运维者与后续实现自动更新的开发者。

**非目标：** 多租户 SaaS 自动滚动发布；服务端进程内静默从 GitHub 覆盖自身。

---

## 一、结论摘要

| 组件 | 推荐机制 | 用户体验 | 与现状关系 |
|------|----------|----------|------------|
| **Latte** | **Velopack** + GitHub Releases | 启动/手动检查 → 弹窗确认 → 下载安装 → 重启 | 在现有 `dotnet publish -r win-x64` 之上打更新包 |
| **Server** | **升级脚本**（备份 → 停服 → 校验哈希 → 替换 → 启服） | SSH/本地执行；管理页仅「检查更新」提示（可选） | 固化并增强部署文档 §9.2，复用 `backup-bisheng.sh` |

原则：

1. 安装包/压缩包校验 **SHA-256**（后续可加 Authenticode / 发布者签名）
2. 客户端**不静默升级**（个人笔记需用户确认）
3. 服务端**不**由 Kestrel 进程自下载自替换
4. 客户端与服务端版本**可独立升级**；破坏性变更用 `minClient` / `minServer` 声明

## 二、版本与发布通道

### 2.1 版本号

采用 **SemVer**（`MAJOR.MINOR.PATCH`），与程序集信息一致：

| 产物 | 版本来源 |
|------|----------|
| Latte / Server | 根目录 `Directory.Build.props` 的 **`BiShengVersion`**（两端 csproj 引用同一值） |
| 共享协议 | `BiSheng.Shared` 可单独标注；破坏协议时抬 Server `minClient` |

Git tag：**单一** `v1.2.3`，一个 GitHub Release 同时挂 Latte（Velopack）与 Server（zip）。

### 2.2 发布通道：GitHub Releases

仓库：`https://github.com/hillsburg/BiSheng`（以实际 remote 为准）。

每个 Release 建议资产（命名约定）：

```text
BiSheng-Latte-{version}-win-x64-Setup.exe     # Velopack 安装/更新入口（或等价产物）
BiSheng-Latte-{version}-win-x64.nupkg         # Velopack 包（若工具链要求）
BiSheng.Server-{version}-linux-x64.zip        # 对应 dotnet publish 产出（非自包含）
SHA256SUMS.txt                                # 各文件 sha256
update-manifest.json                          # 见 §2.3（可同时作为 latest 指针）
```

> Server 当前部署文档使用 **framework-dependent**（`--self-contained false`），目标机需已装 .NET 8 ASP.NET Core Runtime。更新包保持同一模式，避免与文档分叉。

### 2.3 清单文件 `update-manifest.json`

放在 Release 资产中，并可用「最新稳定」固定 URL（例如 Release 的最新资产，或 `docs`/GitHub Pages 上的 `latest-update-manifest.json` 指向当前版本）。

```json
{
  "schemaVersion": 1,
  "channel": "stable",
  "publishedAt": "2026-07-18T00:00:00Z",
  "latte": {
    "version": "1.2.3",
    "minOs": "Windows 10",
    "feedUrl": "https://github.com/hillsburg/BiSheng/releases/download/v1.2.3/",
    "setupFile": "BiSheng-Latte-1.2.3-win-x64-Setup.exe",
    "sha256": "…",
    "releaseNotesUrl": "https://github.com/hillsburg/BiSheng/releases/tag/v1.2.3",
    "minServer": "1.0.0"
  },
  "server": {
    "version": "1.2.3",
    "runtime": "net8.0",
    "rid": "linux-x64",
    "packageFile": "BiSheng.Server-1.2.3-linux-x64.zip",
    "downloadUrl": "https://github.com/hillsburg/BiSheng/releases/download/v1.2.3/BiSheng.Server-1.2.3-linux-x64.zip",
    "sha256": "…",
    "minClient": "1.0.0",
    "releaseNotesUrl": "https://github.com/hillsburg/BiSheng/releases/tag/v1.2.3"
  }
}
```

- `minServer`：此 Latte 要求的最低服务端版本（可选，用于连接时提示）
- `minClient`：此 Server 要求的最低客户端版本（可选）

## 三、服务端升级设计（对齐现有 publish / `/opt/bisheng`）

### 3.1 现有流程（部署文档）

构建：

```bash
cd src/BiSheng.Server
dotnet publish -c Release -o ./publish --self-contained false
```

首次部署：上传至 `/opt/bisheng/`，保留后续生成的 `bisheng.db`、`uploads/`、`appsettings.Production.json`、`data-protection-keys/`。

手动更新（§9.2）：停服 → `backup-bisheng.sh` → 拷贝 dll/json（不覆盖 Production）→ 启服。

### 3.2 目标流程：`upgrade-bisheng.sh`

路径：仓库 [`scripts/server-upgrade/upgrade-bisheng.sh`](../../scripts/server-upgrade/)。

```text
输入：
  PACKAGE=/path/to/BiSheng.Server-{ver}-linux-x64.zip
  或 DOWNLOAD_URL=… + EXPECTED_SHA256=…
  BISHENG_HOME=/opt/bisheng（默认）

步骤：
  1. 预检：systemd 单元存在；sqlite3 / unzip 可用；磁盘空间
  2. 备份：调用 backup-bisheng.sh（升级前必须成功）
  3. systemctl stop bisheng
  4. 解压到临时目录 $BISHENG_HOME/.stage-{stamp}/
  5. 校验 zip 的 sha256（与 SHA256SUMS 或参数一致）
  6. 保留名单（不得被覆盖删除）：
       bisheng.db*、uploads/、appsettings.Production.json、
       data-protection-keys/、backup/
  7. 将 stage 内程序文件同步到 $BISHENG_HOME
       （覆盖 *.dll / 框架 json / content files；跳过保留名单）
  8. 可选：dotnet BiSheng.Server.dll 相关迁移
       （若使用 EF：dotnet ef database update，或应用启动自动 Migrate——以代码实际为准）
  9. chown -R bisheng:bisheng $BISHENG_HOME（除策略另定外）
 10. systemctl start bisheng
 11. 健康检查：curl -sf http://127.0.0.1:8090/admin/login （或日后 /health）
 12. 失败：systemctl stop；从升级前备份恢复程序文件策略见 §3.4；必要时 restore-bisheng.sh
```

环境变量建议：

| 变量 | 默认 | 说明 |
|------|------|------|
| `BISHENG_HOME` | `/opt/bisheng` | 安装根 |
| `PACKAGE` | （必填之一） | 本地 zip 路径 |
| `DOWNLOAD_URL` | （必填之一） | 远程 zip |
| `EXPECTED_SHA256` | （强烈建议） | 包哈希 |
| `SKIP_BACKUP` | `0` | 仅灾难恢复调试用，生产禁止 |

### 3.3 与 `dotnet publish` 产物的对应关系

开发机或 CI：

```bash
cd src/BiSheng.Server
dotnet publish -c Release -o ./publish --self-contained false
# 可选：-r linux-x64 若需 RID 特定原生依赖；当前文档未强制 RID
cd publish
zip -r ../BiSheng.Server-{version}-linux-x64.zip .
sha256sum ../BiSheng.Server-{version}-linux-x64.zip >> ../SHA256SUMS.txt
```

上传到 GitHub Release 后，服务器：

```bash
sudo BISHENG_HOME=/opt/bisheng EXPECTED_SHA256=… \
  /usr/local/bin/upgrade-bisheng.sh /tmp/BiSheng.Server-1.2.3-linux-x64.zip
```

或继续用现有 §9.2 手工 `scp` + `cp`（脚本只是把该流程标准化）。

### 3.4 回滚

| 失败点 | 动作 |
|--------|------|
| 备份失败 | 中止，不替换 |
| 哈希不符 | 中止，不替换 |
| 启服失败 / 健康检查失败 | 停服；用升级前备份的程序快照（建议 upgrade 在替换前把旧 dll 打成 `pre-upgrade-{stamp}.tar.gz`）还原；DB 一般不回滚除非迁移破坏数据 |
| 迁移破坏数据 | `restore-bisheng.sh` 指向升级前 `backup/` 目录（见备份指南） |

### 3.5 管理页「检查更新」（可选，P2）

- 端点示例：`GET /admin/api/update-check`（需管理员 Cookie）或只读配置页展示
- 服务端请求可配置的 `Update:ManifestUrl`（默认 GitHub raw/release 上的 `update-manifest.json`）
- **仅展示**「当前版本 / 最新版本 / 下载链接 / 发布说明」，**不**在服务器上下载执行
- 真正升级仍由运维跑 `upgrade-bisheng.sh`

## 四、Latte 客户端升级设计（Velopack）

### 4.1 现有发布

```bash
dotnet publish src/BiSheng.Latte/BiSheng.Latte.csproj -c Release -r win-x64 --self-contained false
```

产出：`src/BiSheng.Latte/bin/Release/net8.0-windows/win-x64/publish/`。

目标机需 [.NET 8 Desktop Runtime](https://dotnet.microsoft.com/download/dotnet/8.0)（与 README 一致）。若希望「免装 Runtime」，可另发 `--self-contained true` 通道（体积更大）；首版建议与 README 一致 **framework-dependent**。

### 4.2 为什么用 Velopack

- Squirrel.Windows 事实继任者，.NET / WPF 生态成熟
- 支持安装、差分/全量更新、重启应用
- 可对接 GitHub Releases 作为 feed

### 4.3 用户流程

```text
启动 Latte（或「帮助 → 检查更新」）
  → Velopack 查询 feed / update-manifest
  → 无新版本：静默或状态栏「已是最新」
  → 有新版本：对话框
        标题：发现新版本 {version}
        正文：摘录 release notes 或打开浏览器
        [稍后提醒] [立即更新]
  → 立即更新：下载 → 校验 → 安装 → 退出并重启
```

约束：

- **默认不自动下载安装**（可设置「仅 Wi‑Fi 时检查」等，非必须）
- 更新前尽量：退出前 Flush 同步（已有 `FlushOnExit`）；可选提示「请先完成同步」
- 更新源仅 HTTPS

### 4.4 工程接入要点（实现阶段）

1. 引入 Velopack 包与启动时 `UpdateManager`（伪代码级）
2. CI 或本地脚本：`dotnet publish` → `vpk pack`（以 Velopack CLI 当时文档为准）→ 上传 Release
3. 关于页 / 外观或同步设置旁展示 **当前版本号**
4. 首次从「绿色解压 publish 目录」迁移到「Velopack 安装版」：用户数据在 `%LocalAppData%\BiSheng\Latte\`，与安装目录分离；首次启动会把 exe 旁旧的 `local.db` / `config.json` / `images` 自动迁入（源文件改名为 `*.bak-before-appdata`）。

> **配置路径：** `local.db`、`config.json`、`images/`、`log/` 已固定到 `%LocalAppData%\BiSheng\Latte\`（见 `LatteAppPaths`）。

### 4.5 与服务端版本提示（可选）

Latte 连接成功后可读服务端某个轻量字段（例如管理 API 或现有 verify 扩展）中的 `serverVersion` + `minClient`：

- 客户端过旧 → 提示升级 Latte  
- 服务端过旧 → 提示运维升级 Server（链到部署文档 / Release）

首期可不做协议字段，仅靠 Release 说明。

## 五、CI / 发版流水线（建议）

在现有 [`.github/workflows/ci.yml`](../../.github/workflows/ci.yml)（仅 test）之外，已增加 [`.github/workflows/release.yml`](../../.github/workflows/release.yml)：

```text
on:
  push:
    tags: ['v*']
  workflow_dispatch:  # 输入版本号

  - test
  - release：scripts/release/pack-all.ps1 → gh release create（Latte + Server + SHA256SUMS）
```

本地也可：`.\scripts\release\pack-all.ps1` 后手工上传。

## 八、风险与缓解

| 风险 | 缓解 |
|------|------|
| 升级中断半套 dll | stage 目录 + 启服前原子替换；失败回滚 `pre-upgrade-*.tar.gz` |
| 覆盖 `appsettings.Production.json` | 保留名单；脚本显式跳过 |
| DP keys 丢失导致 Cookie 失效 | 备份包含 `data-protection-keys/`；升级不删该目录 |
| 假更新包 | HTTPS + SHA-256；后续代码签名 |
| Velopack 覆盖用户数据 | 数据目录与安装目录分离（LocalAppData） |
| 协议不兼容 | manifest `minClient` / `minServer` + Release 说明 |
