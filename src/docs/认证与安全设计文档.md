# 认证与安全设计文档

## 概述

BiSheng Server 采用**双认证方案**，根据访问者身份分别保护不同入口：

| 认证方式 | 适用场景 | 保护范围 |
|----------|----------|----------|
| **Cookie** | 管理员浏览器访问 | Razor Pages（`/admin/*`） |
| **API Key** | 客户端程序调用 | Web API（`/api/*`） |

单用户模式：系统仅有一个管理员账户，所有 API Key 均绑定同一用户。

---

## 首次初始化流程

```
浏览器首次访问任意路径
    │
    ▼
SetupMiddleware 检测 ServerConfigs.IsSetup = false
    │
    ├── 非 /admin/setup 路径 → 强制重定向到 /admin/setup
    │
    └── /admin/setup → 显示初始化表单
            │
            ▼ (POST)
        创建管理员用户（PBKDF2 哈希密码）
        生成首个 API Key（SHA256 哈希存储）
        标记 IsSetup = true
        重定向到 /admin/login?setup=done
```

**SetupMiddleware** 中间件位于认证管道之前，确保未完成初始化时，除 `/admin/setup` 外的所有请求都被拦截。

## Cookie 认证（管理面板）

### 配置

```csharp
.AddCookie(CookieAuthenticationDefaults.AuthenticationScheme, options =>
{
    options.LoginPath = "/admin/login";
    options.LogoutPath = "/admin/logout";
    options.Cookie.Name = "bisheng.admin";
    options.Cookie.HttpOnly = true;                       // 禁止 JS 访问
    options.Cookie.SecurePolicy = CookieSecurePolicy.SameAsRequest;  // HTTPS 时自动 Secure
    options.Cookie.SameSite = SameSiteMode.Lax;           // 防 CSRF
    options.ExpireTimeSpan = TimeSpan.FromDays(7);
    options.SlidingExpiration = true;                      // 活跃时自动续期
});
```

### 登录流程

```
POST /admin/login (含 AntiForgeryToken)
    │
    ├── 速率检查：每 IP 每分钟 ≤ 5 次（超限重定向登录页）
    │
    ├── CSRF 检查：验证 Anti-Forgery Token
    │
    ├── 若存在有效的 pending-login Cookie（短时、Data Protection 加密）
    │   └── 第二步：仅用 Cookie 内 UserId 校验 TOTP → 成功则签发管理 Cookie
    │
    ├── 否则第一步：查 Users 表 → 密码验证（PBKDF2）
    │   ├── 失败 → 返回"用户名或密码错误"（不区分用户/密码）
    │   ├── 开发环境 → 跳过 TOTP，直接签发 ClaimsIdentity
    │   └── 生产环境 → 写入 pending-login Cookie（约 5 分钟），进入 TOTP 步
    │
    ▼
ClaimsPrincipal 包含：
    - NameIdentifier = UserId (Guid)
    - Name = Username
    - Role = "Admin"
```

> **安全要点：** 第二步不信任客户端提交的 `PendingUserId`。伪造 GUID 无法绕过密码校验。  
> 初始化 `/admin/setup` 同理：TOTP 密钥与密码存于 `pending-setup` 加密 Cookie，不回写到 hidden 字段。  
> 实现：`AdminPendingSessionService`（Data Protection）；Cookie `bisheng.admin.pending-login`（5 分钟）、`bisheng.admin.pending-setup`（15 分钟），均为 HttpOnly。

### 保护范围

所有 `[Authorize]` 标注的 Razor Pages（使用默认 Cookie 方案）：

- `Index`（仪表盘）
- `Notes`（笔记管理）
- `Folders`（文件夹管理）
- `Keys`（API Key 管理）

未认证访问自动重定向到 `/admin/login`。

## API Key 认证（客户端 API）

### 传递方式

| 位置 | 格式 | 适用场景 |
|------|------|----------|
| HTTP Header | `X-Api-Key: {64字符hex}` | 所有 REST `/api/*` 调用 |
| Query String | `?access_token={64字符hex}` | 仅 SignalR `/hubs/*`（WebSocket） |
| Authorization | `Bearer {64字符hex}` | 仅 SignalR Hub（长轮询等） |

> REST 路径**不再**接受 `?api_key=` query，避免访问日志泄露。详见 [P2组合根与API安全设计文档](./P2组合根与API安全设计文档.md)。

### 验证流程

```
请求到达 → ApiKeyAuthHandler.HandleAuthenticateAsync()
    │
    ├── 提取 Key：REST 仅 Header；Hub 允许 Header / access_token / Bearer
    ├── 空 → NoResult（401）
    │
    ├── 计算 SHA256(Key) → 查 ApiKeys 表匹配哈希
    │   ├── 命中且 IsActive → 签发 Claims（含 UserId、Username、DeviceName）
    │   └── 未命中或已吊销 → Fail（401）
    │
    └── 成功 → 返回认证 Ticket
```

### 存储方式

- 存储格式：`SHA256(原始Key)` → 64 字符 hex 小写
- 数据库仅存哈希值，不保存原始 Key

### 保护范围

所有 `[Authorize(AuthenticationSchemes = ApiKeyAuthHandler.SchemeName)]` 标注的 Controller：

| Controller | 路由前缀 | 功能 |
|------------|----------|------|
| `AuthController` | `/api/auth` | Key 有效性验证 |
| `NotesController` | `/api/notes` | 笔记 CRUD |
| `FoldersController` | `/api/folders` | 文件夹 CRUD |
| `ImagesController` | `/api/images` | 图片上传/下载/增量拉取（唯一附件 API；旧 `/api/files` 已废弃） |
| `SyncController` | `/api/sync` | 增量同步 + SignalR Hub |

## 密码存储

### PBKDF2

使用 ASP.NET Identity 内置 `PasswordHasher<TUser>`：

- **算法**：PBKDF2 + HMAC-SHA256
- **迭代**：10000 次（框架默认）
- **Salt**：随机 128-bit
- **输出格式**：v3 标识前缀 `AQAAAA...`，包含算法版本、迭代次数、salt、hash

```csharp
internal static string HashPassword(string password)
{
    return PasswordHasher.HashPassword(null!, password);
}

internal static bool VerifyPassword(string password, string storedHash)
{
    var result = PasswordHasher.VerifyHashedPassword(null!, storedHash, password);
    return result == PasswordVerificationResult.Success
        || result == PasswordVerificationResult.SuccessRehashNeeded;
}
```

## 安全防护措施

### 1. CSRF 防护

**Razor Pages（表单提交）：**
- 所有 `<form method="post">` 均包含 `@Html.AntiForgeryToken()`
- 框架自动验证 Token，不匹配则拒绝请求

**Cookie 层：**
- `SameSite = Lax`：跨站 POST 请求不携带 Cookie
- 双重保护：Token + SameSite

### 2. HTTPS 强制

HTTPS 终结和强制由反向代理层（Apache）负责，应用层不使用 `UseHttpsRedirection()` 和 `UseHsts()`，避免与代理层重复导致重定向循环。

Apache 层配置：
- 端口 80 自动 301 重定向到 443（CertBot 配置）
- HSTS Header 由 Apache 注入（`Strict-Transport-Security: max-age=31536000`）

### 3. 登录暴力破解防护

使用 ASP.NET 内置 Rate Limiting，**仅对登录 POST** 按 IP 分区限流（GET 加载页面不计入）：

```csharp
options.AddPolicy("login", context =>
{
    if (!HttpMethods.IsPost(context.Request.Method))
        return RateLimitPartition.GetNoLimiter(...);

    var ip = context.Connection.RemoteIpAddress?.ToString() ?? "unknown";
    return RateLimitPartition.GetFixedWindowLimiter(ip, _ => new FixedWindowRateLimiterOptions
    {
        PermitLimit = 5,
        Window = TimeSpan.FromMinutes(1),
        QueueLimit = 0,
    });
});
```

| 参数 | 值 | 说明 |
|------|----|------|
| 范围 | POST `/admin/login` | GET 刷新页面不受限 |
| 窗口 | 1 分钟 | 固定时间窗口 |
| 上限 | 5 次/IP | 超过后返回 429 |
| 队列 | 0 | 不排队，直接拒绝 |

通过 `[EnableRateLimiting("login")]` 标注在 `LoginModel` 上生效。触发 429 时记录 Warning 日志（`BiSheng.Server.RateLimiting`），并重定向至 `/admin/login?rateLimited=1`，在登录页展示「请约 1 分钟后再试」提示。

未设置全局 `GlobalLimiter`，因此**只有显式标注了该策略的端点才会被限流**。

### 3.2 限流范围

| 路径 / 功能 | 是否限流 | 说明 |
|-------------|----------|------|
| POST `/admin/login` | **是** | 密码 / TOTP 提交，5 次/分钟/IP |
| GET `/admin/login` | 否 | 打开或刷新登录页 |
| `/api/sync`（Pull / Push） | **否** | `SyncController` 无 `[EnableRateLimiting]` |
| `/hubs/sync`（SignalR） | **否** | `SyncHub` 无限流标注 |
| `/api/images`、`/api/notes` 等 API | **否** | 客户端 API Key 调用不受此策略约束 |
| 其它 Razor Pages（`/admin/*`） | 否 | 除 Login 页外均未绑定 `login` 策略 |

**笔记同步、图片上传、API Key 验证等客户端高频操作均不会被 429。** 限流仅用于管理后台登录防暴力破解，不影响 Latte 客户端的正常同步行为。

`UseRateLimiter()` 虽在管道中全局注册，但未绑定 `login` 策略的请求会直接放行，不会计数。

### 3.3 反向代理与真实 IP

生产环境 `ForwardedHeaders` **仅信任本机反代**（`127.0.0.1` / `::1`），以便从 Apache 传入的 `X-Forwarded-For` 解析真实客户端 IP，供限流与登录失败日志使用。开发环境不添加 KnownProxy，直连 Kestrel 时不采纳 Forwarded 头，防止伪造 IP。

Apache VirtualHost、Systemd 环境变量与验证步骤见 [反向代理与真实IP传递设计文档.md](./反向代理与真实IP传递设计文档.md) 与 [部署文档.md](./部署文档.md)。

### 4. Cookie 安全属性

| 属性 | 值 | 防护 |
|------|----|------|
| `HttpOnly` | `true` | 阻止 XSS 窃取 Cookie |
| `SecurePolicy` | `SameAsRequest` | HTTPS 时自动设 Secure |
| `SameSite` | `Lax` | 阻止跨站 CSRF |
| `ExpireTimeSpan` | 7 天 | 限制会话有效期 |
| `SlidingExpiration` | `true` | 活跃时自动续期 |

### 5. API Key 安全

- **哈希存储**：数据库仅存 SHA256，泄露数据库不暴露原始 Key
- **传输方式**：Header 优先，Query 仅用于 WebSocket

### 6. 信息泄露防护

- 登录失败统一返回"用户名或密码错误"（不区分用户不存在/密码错误）
- API Key 仅在创建时显示一次（通过 TempData），刷新后不可再查看
- Swagger 仅在 Development 环境启用

## 关键设计决策

| 决策 | 选择 | 原因 |
|------|------|------|
| 密码哈希 | PBKDF2（Identity 内置） | 无需额外依赖，抗暴力破解，支持未来自动升级迭代次数 |
| API Key 哈希 | SHA256 | Key 本身 256-bit 随机，无需慢哈希（非用户可记忆密码） |
| Cookie 方案 | ASP.NET Core Cookie Auth | 轻量，无需 JWT Token 管理 |
| Rate Limiting | 内置中间件 + IP 分区 | 零依赖，单用户场景下足够 |

## 中间件管道顺序

```
请求进入
  │
  ├── UseForwardedHeaders()       识别真实 IP 和协议（反代场景）
  ├── UseRateLimiter()             请求速率限制
  ├── UseCors()                    跨域处理
  ├── SetupMiddleware              首次访问强制 /admin/setup
  ├── UseAuthentication()          解析 Cookie / API Key → 填充 User
  ├── UseAuthorization()           检查 [Authorize] 权限
  │
  ├── MapControllers()             API 路由
  ├── MapRazorPages()              页面路由
  └── MapHub<SyncHub>("/hubs/sync") SignalR Hub
```

管道顺序至关重要：ForwardedHeaders 在最前（确保后续中间件获取真实 IP）；Rate Limiter 在认证之前，且**仅对标注了 `[EnableRateLimiting("login")]` 的登录 POST 生效**（同步 API 不限流）；Setup 中间件在认证之前（确保未初始化时拦截所有请求）。
