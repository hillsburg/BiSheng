## 二、P0-A：服务端统一写路径

### 2.1 问题（重构前）

```
REST (NotesController / FoldersController / NoteRevisionsController)
  → 各自 ReserveNextVersionAsync + 写实体 + SyncLog + SignalR

Sync Push (SyncService.PushAsync)
  → ApplyClientChange（独立校验与写入逻辑）
```

两条路径重复实现 Folder 拓扑、FK、级联软删、NoteRevision 等，任何一侧修复 bug 另一侧可能遗漏。

### 2.2 目标架构

```
HTTP 层
  NotesController / FoldersController / NoteRevisionsController
  SyncController
        ↓
编排层（开事务、SaveChanges、Commit、HTTP 映射）
  INoteMutationService / IFolderMutationService
  ISyncService.PushAsync
        ↓
写内核（校验通过后才 ReserveVersion + ChangeTracker，不 Save / 不 Notify）
  IEntityChangeWriter (EntityChangeWriter)
        ↓
基础设施
  UserSyncVersionService、NoteRevisionService、FolderGraphValidator
        ↓
Commit 之后
  ISyncChangeNotifier → SignalR（轻量 ChangeDto，无 Payload）
```

**契约**：

- `IEntityChangeWriter.TryApplyAsync` 在**调用方已 `BeginTransaction` 的 DbContext** 上运行；只改 ChangeTracker，**不** `SaveChanges`，**不**发 SignalR。
- REST 经 `*MutationService` 包事务；Push 在 `SyncService` 单事务内循环调用 Writer，整批成功后 `NotifyBatchAsync`。
- 校验失败返回 `MutationSkipped`，**不消耗版本号**（与 Push `FailedEntityIds`、REST 400 一致）。

### 2.3 核心类型

路径：`BiSheng.Server/Services/Mutations/`

| 类型 | 职责 |
|------|------|
| `EntityMutation` | 与 `ClientChangeDto` / REST 等价的内部变更描述（EntityType、EntityId、Action、Payload、UpdatedAt） |
| `MutationBatchContext` | Push 批次共享上下文（如 `AvailableFolderIds`，供本批新建 Folder 的 FK 校验） |
| `MutationWriteOptions` | 写入选项：`RecordNoteRevision`；`ForceNoteRevision`（恢复历史仅 hash 去重） |
| `MutationApplyResult` | `MutationApplied(AppliedMutation)` 或 `MutationSkipped(reason)` |
| `AppliedMutation` | 成功结果：根变更 Version、Payload、级联 `CascadeAppliedMutation` 列表 |
| `IEntityChangeWriter` | 唯一写内核接口 |
| `EntityChangeWriter` | 自原 `SyncService.ApplyClientChange` 抽出，REST / Push 共用 |
| `INoteMutationService` / `IFolderMutationService` | REST 编排：校验 → 事务 → Writer → Save → Commit → Notify |
| `ISyncChangeNotifier` / `SyncChangeNotifier` | Commit 后推送轻量 SignalR 通知（无 Payload） |
| `ClientChangeDtoExtensions` | DTO → `EntityMutation` 映射 |
| `SyncChangeDtoBuilder` | Pull / REST 响应构建完整 `ChangeDto`（含 Payload） |

### 2.4 Push 批次流程

```csharp
await using var tx = await _db.Database.BeginTransactionAsync(ct);
var batchContext = new MutationBatchContext { AvailableFolderIds = ... };

foreach (var change in orderedChanges)
{
    var mutation = change.ToEntityMutation();
    var result = await _entityChangeWriter.TryApplyAsync(
        _db, userId, mutation, batchContext, options, ct);

    switch (result)
    {
        case MutationApplied applied:
            appliedList.Add(applied.Applied);
            // 更新 batchContext.AvailableFolderIds（Create Folder 等）
            break;
        case MutationSkipped:
            failedEntityIds.Add(change.EntityId);
            break;
    }
}

await _db.SaveChangesAsync(ct);
await tx.CommitAsync(ct);
await _notifier.NotifyBatchAsync(userId, appliedList, ct);
```

版本仍采用 **per-change 原子分配**（校验通过后再 `ReserveNextVersionAsync`），与 [§3.3](./版本管理与增量同步设计文档.md) 一致。

### 2.5 REST 流程（以创建笔记为例）

```csharp
// NoteMutationService.CreateAsync
var mutation = new EntityMutation { ... };
await using var tx = await _db.Database.BeginTransactionAsync(ct);
var result = await _writer.TryApplyAsync(_db, userId, mutation, batchContext: null, options, ct);
if (result is MutationSkipped skip) return BadRequest(skip.Reason);
await _db.SaveChangesAsync(ct);
await tx.CommitAsync(ct);
await _notifier.NotifyAppliedAsync(userId, ((MutationApplied)result).Applied, ct);
```

`NoteRevisionsController.RestoreRevision` 委托 `INoteMutationService.RestoreFromRevisionAsync`，恢复正文同样经 Writer + Notify，保证 SyncLog / 版本 / SignalR 与 Push 一致。

### 2.6 依赖注入

`ServiceCollectionExtensions.AddBiShengServerCore()` 注册：

| 服务 | 生命周期 |
|------|----------|
| `IEntityChangeWriter` → `EntityChangeWriter` | Scoped |
| `ISyncChangeNotifier` → `SyncChangeNotifier` | Scoped |
| `INoteMutationService` → `NoteMutationService` | Scoped |
| `IFolderMutationService` → `FolderMutationService` | Scoped |

`ISyncService` 注入 `IEntityChangeWriter` 与 `ISyncChangeNotifier`；Controller 仅注入 `*MutationService` 或 `ISyncService`。

### 2.8 行为一致性保证

`RestSyncParityTests` 验证同 payload 下：

| 场景 | REST | Push | 一致点 |
|------|------|------|--------|
| 创建 Note | 200 + SyncLog | Success + SyncLog | Action / Payload 形态等价 |
| 无效 FolderId | 400 | `FailedEntityIds` | 均不消耗 `CurrentVersion` |
| 删除 Folder | 级联软删 | 级联软删 | SyncLog 条数与版本序列一致 |
| 更新 Note | SyncLog Update | SyncLog Update | 版本递增与实体字段一致 |
