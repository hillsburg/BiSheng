# 同步与安全设置使用指南

本文说明 BiSheng Latte 工具栏 **「同步与安全」** 对话框如何配置。改完后点 **保存** 即可在当前进程生效，**无需重启应用或服务器**（详见文末「何时生效」）。

打开方式：主窗口工具栏 → **同步与安全**。

对话框含三个页签：**连接** / **同步** / **数据安全**。顶部会显示当前连接状态。

---

## 快速上手：三种常见场景

### 1. 只用本机（不连服务器）

| 项 | 建议 |
|----|------|
| 服务器地址 / API Key | **留空** |
| 数据安全 | 保持默认（标准档位、自动备份、退出备份） |

无需配置同步页签。笔记存在应用目录下的 `local.db`。

### 2. 连接自托管服务器（推荐默认）

1. 部署并启动 [BiSheng.Server](./部署文档.md)，浏览器完成首次设置。
2. 打开管理面板 **`/admin/keys`**，创建 API Key（创建时只显示一次，请妥善保存）。
3. 在 **连接** 页签填写：
   - **服务器地址**：如 `http://localhost:8090` 或 `https://your-host`
   - **API Key**：上一步拿到的密钥
4. 点 **测试连接**，确认成功。
5. 勾选 **启用同步**（填齐地址与 Key 后才会显示）。
6. 点 **保存**。

同步页签与数据安全页签一般可保持默认。

### 3. 更怕丢数据（保守模式）

在 **数据安全** 页签将 **安全档位** 选为 **保守（更密 Push / 备份 / 历史）**，再点保存。

保存时会自动收紧部分参数（Push 约 15 秒、备份间隔 12 小时、保留份数至少 21、强制开启编辑日志），并使本地笔记历史采样更密。详见下文「安全档位」。

> 注意：选择保守模式后，你在「同步」页签里手填的 Push 周期可能被覆盖。

## 连接

配置服务器凭据；关闭「启用同步」可暂停所有同步，开启后可手动同步或配合自动同步选项。

| 设置 | 说明 | 默认 |
|------|------|------|
| **服务器地址** | 同步服务器根地址。留空（或 API Key 为空）→ 纯本地模式，不同步 | 空 |
| **API Key** | 客户端认证密钥，请求头 `X-Api-Key`。从 `/admin/keys` 获取 | 空 |
| **启用同步** | 关闭后暂停 Push / Pull 与工具栏「手动同步」；凭据仍保留 | 有凭据时默认开启 |
| **测试连接** | 探测 `/api/auth/verify-key`，不替代「保存」 | — |

要点：

- **地址与 API Key 修改后必须点保存**；**启用同步开关会立刻写入配置**。
- API Key 以 Windows DPAPI（当前用户）加密保存在应用目录的 `config.json`。
- 仅测试**已保存**的凭据时，成功/失败会反映到连接状态；测试尚未保存的输入框内容不会改已保存状态。
- 生产环境建议使用 HTTPS，见 [部署文档](./部署文档.md)、[认证与安全设计文档](./认证与安全设计文档.md)。

### 暂停同步但保留凭据

填好地址与 Key 后，关闭 **启用同步**：状态会显示「同步已关闭」，本地照常编辑，不再 Push / Pull，直到重新开启。

## 同步

单人多端：本地编辑先落盘，按节奏 Push 到云端；其他设备的变更通过 SignalR **实时 Pull**。本页参数在已启用同步且能连上服务器时才真正起作用。

| 设置 | 说明 | 默认 | 范围 |
|------|------|------|------|
| **笔记 Push 周期（秒）** | 有待推送变更时，按此间隔上传 | 30 | 5–600 |
| **关闭应用前推送未同步变更** | 退出前尽量把未推送变更上传（强烈建议开启） | 开 | — |
| **窗口重新激活时同步** | 窗口从后台回到前台时 Push + Pull（适合笔记本休眠恢复） | 开 | — |
| **网络 / SignalR 恢复时同步** | 网络或实时通道恢复后 Push + Pull | 开 | — |
| **图片上传周期（秒）** | 扫描并上传未同步的本地图片 | 10 | 5–600 |
| **图片拉取周期（秒）** | 从服务器增量下载新图片 | 60 | 10–3600 |

说明：

- Push 周期越短，崩溃时「尚未推上去」的窗口越小，网络与电量开销略增。
- 图片周期与 [图片功能使用指南](./图片功能使用指南.md) 一致；打开含未同步图片的笔记时也会按需触发下载。
- 节电或按流量计费网络时，可将 Push 调到 60–120 秒、图片拉取调大，但请保持 **关闭应用前推送** 开启。

## 数据安全

自动备份、编辑日志、回收站与导出提醒。设计背景见 [数据安全策略设计文档](./数据安全策略设计文档.md)。

### 安全档位

| 档位 | 行为 |
|------|------|
| **标准（默认）** | 默认 Push / 备份 / 历史采样节奏 |
| **保守（更密 Push / 备份 / 历史）** | 保存时强制：Push **15** 秒、退出前推送开启、备份间隔 **12** 小时、保留份数至少 **21**、编辑日志开启；笔记历史自动快照更密 |

### 本地库备份

| 设置 | 说明 | 默认 | 范围 |
|------|------|------|------|
| **启用 local.db 自动备份** | 总开关；关闭后不做定时与退出备份 | 开 | — |
| **关闭应用时备份** | 退出时备份（在落盘之后、停止同步之前） | 开 | — |
| **保留备份份数** | 超出后删除最旧备份 | 14 | 3–90 |
| **定时备份间隔（小时）** | `0` = 仅退出时备份；否则在启动等时机按间隔检查 | 24 | 0–168 |
| **使用默认位置** | 默认目录见下方；也可「浏览…」自定义 | 开 | — |
| **管理备份…** | 打开备份管理窗口，查看 / 恢复 | — | — |

路径：

- 运行库：`{应用目录}/local.db`
- 默认备份：`%LocalAppData%\BiSheng\Latte\backups\local-yyyyMMdd-HHmmss.db`（及同名 `.meta.json`）

建议：自定义备份目录尽量选**非应用安装目录**的磁盘位置，避免卸载或清空程序目录时一并删掉备份。

### 编辑日志、回收站、导出提醒

| 设置 | 说明 | 默认 | 范围 |
|------|------|------|------|
| **记录只追加编辑日志** | 本地只追加操作记录与内容哈希（**不能据此恢复正文**），Push 成功后打标，便于排查 | 开 | — |
| **编辑日志保留（天）** | 超过天数的日志会被清理 | 30 | 7–365 |
| **回收站保留（天）** | 软删除笔记在本机保留多久后硬删；可在工具栏「回收站」恢复 | 30 | 7–365 |
| **提醒定期导出全库** | 启动时若超过间隔未做全库导出则提示 | 开 | — |
| **导出提醒间隔（天）** | 距上次全库导出的天数阈值 | 7 | 1–90 |

全库导出（BiSheng Archive）可在工具栏相关入口执行，作为异地副本，与自动备份互补。

## 配置文件位置

| 内容 | 路径 |
|------|------|
| 服务器地址、API Key、是否启用同步 | `{应用目录}/config.json` |
| 同步节奏（Push / 图片等） | `%LocalAppData%\BiSheng\Latte\sync.json` |
| 数据安全（备份 / 日志 / 回收站等） | `%LocalAppData%\BiSheng\Latte\data-safety.json` |
| 本地笔记库 | `{应用目录}/local.db` |

一般无需手改这些文件；优先用界面保存。

## 何时生效

| 变更 | 生效方式 |
|------|----------|
| **启用同步** | 开关立即写入；保存或（拨过开关后取消关闭时）会重启同步引擎 |
| **服务器地址 / API Key** | 点 **保存** 后写入，并自动重启同步引擎以新凭据连接 |
| **同步页签各项** | 点 **保存** 后热更新定时器并重启引擎 |
| **安全档位 / 多数数据安全项** | 点 **保存** 后写入；备份、日志、回收站等在下次执行对应逻辑时读配置 |
| **导出提醒** | 下次启动时检查 |
| **定时备份间隔** | 写入后，主要在**启动**与**退出**时按间隔判断，不会在会话中途另起一套独立定时器 |

界面与代码中的「重启」指 **同步引擎**（停止再启动连接），不是退出再开整个 Latte 客户端，也不是重启服务器。

## 常见问题

### 测试连接成功，但状态仍不对？

确认已点 **保存**。仅改输入框不保存时，运行中的引擎仍用旧凭据。

### 「启用同步」看不到？

需同时填写非空的服务器地址与 API Key；纯本地模式下该开关隐藏。

### 选了保守模式，Push 周期怎么变了？

保守档位保存时会覆盖 Push / 备份等相关项，这是预期行为。若要自定义 Push，请改回 **标准** 档位后再调数值。

### 多台电脑怎么同步？

每台客户端填写同一服务器地址，并各自在 `/admin/keys` 创建（或使用允许的）API Key，开启同步即可。冲突时客户端会保留双副本，避免静默覆盖。

### 备份和「笔记历史」有什么区别？

- **自动备份**：整库 `local.db` 快照，用于库损坏、误删大量数据等灾难恢复。
- **笔记历史**：单篇笔记的版本快照，用于误改回滚。

二者不能互相替代。详见 [数据安全策略设计文档](./数据安全策略设计文档.md)、[笔记历史版本设计文档](./笔记历史版本设计文档.md)。

### 关掉自动备份可以吗？

可以，但不建议。至少保留 **关闭应用时备份**，或定期使用全库导出。

## 已知限制（用户须知）

| 限制 | 说明 |
|------|------|
| 编辑日志 | 只存哈希与操作类型，**不能**当笔记内容备份 |
| 图片删除 | **仅服务端 GC**：客户端不传播图片 DELETE；孤儿约 7 天后软删、再约 30 天后硬删。本机 `images/` 不随删引用而清理 |
| Pull 游标 | 服务端在 Pull 响应发出前即推进设备游标，**无客户端 ACK**。极端情况下响应丢失且日志已被裁剪时，可能收到「需全量同步」（非静默丢笔记） |
| 退出同步 | 会尽量排空未推送变更；网络长时间失败时仍可能残留 pending，下次启动会继续推 |
| 全量同步中断 | 若曾出现「全量同步」且异常退出，下次启动会尝试从抢救文件恢复未上云编辑 |
| 生产服务器 | 务必 `ASPNETCORE_ENVIRONMENT=Production`，并由反代终结 TLS；非 Production 会跳过管理员 TOTP |
| 服务端备份 | 应用不内置连续复制；运维须备份 `bisheng.db` + `uploads/`，见 [服务端备份与恢复运维指南](./服务端备份与恢复运维指南.md) |

更完整的威胁模型与分层防护见 [数据安全策略设计文档](./数据安全策略设计文档.md)。
