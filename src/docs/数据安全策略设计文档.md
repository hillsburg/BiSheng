# 数据安全策略设计文档

## 一、目标与范围

BiSheng 采用 **本地优先 + 云端同步** 架构。本文档定义**防数据丢失**的分层策略、已实现机制与运维要求。

| 层次 | 职责 | 本文档 |
|------|------|--------|
| 本地热数据 | 编辑落盘、未 Push 队列 | ✅ |
| 同步收敛 | 多端一致、冲突可恢复 | 引用 [版本管理与增量同步设计文档](./版本管理与增量同步设计文档.md) |
| 笔记历史 | 单篇误改回滚（FIFO 100） | 引用 [笔记历史版本设计文档](./笔记历史版本设计文档.md) |
| 灾难备份 | 库损坏、误删、全量重建 | ✅ |
| 服务端运维 | bisheng.db + uploads | 引用 [部署文档](./部署文档.md) |

> **历史版本 ≠ 全库备份。** SyncLog 是 replication log，会被裁剪；NoteRevision 是 per-note 快照，不覆盖文件夹树与未 Push 数据。

---

## 二、威胁模型（简要）

| 风险 | 典型场景 | 主要防线 |
|------|----------|----------|
| 未 Push 丢失 | 30s 间隔内崩溃、离线编辑 | PendingChanges + 退出 Push + **全量重建抢救** |
| 全量同步清库 | SyncLog 裁剪后 stale 设备回归 | **FullResyncRecovery** |
| 本地库损坏 | 磁盘/程序异常 | WAL + **integrity_check** + **自动 local.db 备份** |
| 静默覆盖 | 多端同时改 | 冲突双副本 + 语义指纹去假冲突 |
| 服务端灾难 | 磁盘故障 | 运维脚本 `sqlite3 .backup` + `uploads/`；可选 Litestream（见 [服务端备份与恢复运维指南](./服务端备份与恢复运维指南.md)） |
| 误删单篇 | 用户操作 | NoteRevision + 软删除 |

## 三、分层防护（已实现）

```
┌─────────────────────────────────────────────────────────┐
│ L4 用户导出（Markdown/Word/PDF）— 人工异地副本           │
├─────────────────────────────────────────────────────────┤
│ L3 服务端 bisheng.db + uploads 定时备份 — 运维脚本/可选 Litestream │
├─────────────────────────────────────────────────────────┤
│ L2 笔记历史 NoteRevision / LocalNoteRevision — 单篇回滚 │
├─────────────────────────────────────────────────────────┤
│ L1 本地自动备份 local-*.db — 本机灾难恢复               │
├─────────────────────────────────────────────────────────┤
│ L0 运行时：WAL + PendingChanges + 全量重建抢救          │
└─────────────────────────────────────────────────────────┘
```

### 3.1 L0：运行时 durability

| 机制 | 说明 |
|------|------|
| SQLite WAL | `journal_mode=WAL`，读写并发，崩溃可恢复 |
| `SaveChangesWithLock` | 应用层写锁，避免 locked |
| `PendingChanges` | 未 Push 变更独立表，Push 成功才清除 |
| 退出 flush | 可配置，关闭前 Push（`SyncSettings.FlushOnExit`） |
| 周期 Push | 默认 30s，有待推送才上传 |

### 3.2 L0+：全量重建抢救（FullResyncRecovery）

当服务端返回 `RequiresFullSync: true` 时，旧逻辑会 **清空 Notes/Folders/PendingChanges** 再 Pull，可能丢失**从未 Push 成功**的本地数据。

**现行为：**

1. **清库前** `FullResyncRecovery.Capture`：快照所有 `PendingChanges` + 对应 Note/Folder 实体  
2. **落盘** `FullResyncRescueStore.Save`：写入 `%LocalAppData%\BiSheng\Latte\full-resync-rescue.json`（原子写临时文件再替换）  
3. 再清空本地 Notes / Folders / PendingChanges  
4. **Pull 后** `ApplyAfterPull`：对每条抢救记录  
   - 用 `SyncPayloadFingerprint` 对比 Pull 结果与抢救 Payload  
   - **不一致或实体缺失** → 从快照恢复实体并重新入队 Pending  
   - **一致** → 视为已收敛，不重复入队  
5. 成功后 `FullResyncRescueStore.Clear`；若 Pull/合并中途失败或进程崩溃，保留文件  
6. **启动同步引擎**时若检测到抢救文件，优先 `PerformFullResyncAsync` 恢复（磁盘快照优先于空库 Capture）  

代码：`FullResyncRecovery.cs`、`FullResyncRescueStore.cs`、`SyncFullResyncRunner.cs`。

### 3.3 L1：本地 automatic backup

| 项 | 默认值 | 配置 |
|----|--------|------|
| 启用 | 是 | `DataSafetySettings.EnableAutoBackup` |
| 退出备份 | 是 | `BackupOnExit` |
| 定时备份 | 24 小时 | `BackupIntervalHours`（0=仅退出） |
| 保留份数 | 14 | `BackupRetentionCount`（3–90） |
| 备份目录 | 默认 LocalAppData | `BackupDirectoryUseDefault` / `BackupDirectory` |
| 备份元数据 | 每次备份写 `.meta.json` | 笔记数、完整性、VACUUM 统计等 |

**路径：**

- 运行库：`{exe目录}/local.db`
- 默认备份：`%LocalAppData%\BiSheng\Latte\backups\local-yyyyMMdd-HHmmss.db` + 同名 `.meta.json`
- 自定义目录：见 `DataSafetySettings.BackupDirectory`

**管理 UI：** 工具栏「本地备份」或 同步设置 → 管理备份…

**实现：** 备份前 VACUUM 源库 + `VACUUM INTO` 紧凑快照；sidecar 元数据与备份同目录。

**触发：**

- 应用启动（满足间隔时）
- 主窗口关闭（ForceSave 之后、StopSync 之前）

配置 UI：**工具栏 → 同步与安全 →「数据安全」** 标签页。  
配置文件：`%LocalAppData%\BiSheng\Latte\data-safety.json`  
用户操作说明见 [同步与安全设置使用指南](./同步与安全设置使用指南.md)。

### 3.4 L1+：启动完整性检查

启动时对 `local.db` 执行 `PRAGMA integrity_check`；失败写 error 日志，便于发现损坏库并改用 backups 目录恢复。

代码：`LocalDbIntegrityChecker.VerifyAndLog()`，`App.OnStartup`。

### 3.5 L2 / L3 / L4

见 [笔记历史版本设计文档](./笔记历史版本设计文档.md)、[部署文档](./部署文档.md) § 备份、客户端导出功能。

### 3.6 P1：产品层增强（用户可感知）

| 机制 | 说明 | 配置 |
|------|------|------|
| **编辑日志** | `EditJournal` 表只追加；Push 成功打 `SyncedAtUtc` | `EnableEditJournal`、`EditJournalRetentionDays` |
| **回收站** | 软删保留 N 天，可恢复；到期本机硬删 | `TrashRetentionDays`；工具栏「回收站」 |
| **全库导出** | `BiSheng-Archive_*/manifest.json` + `notes/` + `images/` | 工具栏「导出全部库」 |
| **导出提醒** | 超过 N 天未全库导出时启动提示 | `EnableExportReminder`、`RemindExportAfterDays` |
| **保守模式** | Push 15s、备份 12h、历史采样更密 | `Profile: Conservative` |

回收站恢复会向服务端发送 `Update`，服务端在 Push 时将 `IsDeleted` 置回 `false`（与客户端 Pull 行为一致）。

### sync.json（与数据安全相关项）

| 字段 | 建议 |
|------|------|
| `FlushOnExit` | 生产环境保持 `true` |
| `PeriodicPushIntervalSeconds` | 30–60；间隔越大未 Push 窗口越长 |

## 五、恢复手册

### 5.1 恢复 local.db（客户端）

1. 完全退出 BiSheng Latte  
2. 备份当前损坏的 `local.db`  
3. 从 `%LocalAppData%\BiSheng\Latte\backups\` 选取最近 `local-*.db`  
4. 复制为 `{exe目录}/local.db`（同时删除 `-wal`/`-shm` 若存在）  
5. 重启应用  

### 5.2 恢复服务端

见 [服务端备份与恢复运维指南](./服务端备份与恢复运维指南.md)（`restore-bisheng.sh` / Litestream restore）与 [部署文档](./部署文档.md) §9.3。

### 5.3 全量同步后笔记「复现」

若日志出现 `全量同步后已恢复 N 条未上云本地变更`：说明抢救生效，Pending 已重新入队，等待 Push 或手动同步即可。

## 六、已知局限与后续

| 局限 | 说明 | 状态 |
|------|------|------|
| 无 Push 窗口 | 崩溃到下次备份间仍有秒级风险 | 退出 Flush 排空、保守模式、local.db 备份；**编辑日志仅存 ContentHash，不能恢复正文** |
| 全量重建窗口 | 清库后 Pull 中断曾可能丢未 Push 数据 | **已缓解**：抢救快照落盘 `%LocalAppData%\BiSheng\Latte\full-resync-rescue.json`，启动时自动恢复 |
| SyncLog 裁剪 | 删日志与抬 floor 不同步曾导致静默缺口 | **已缓解**：同事务提交（`SyncLogCompactor`） |
| 图片独立管道 | 正文与图片可能短暂不一致；**客户端不传播图片 DELETE（设计如此）** | **仅服务端 GC**：`ImageGc:OrphanGraceDays`（默认 7）孤儿软删 + `ImageRetentionDays`（默认 30）硬删；全库导出打包 `images/`；图片拉取失败不推进图片游标 |
| Pull 游标无 ACK | 服务端在 Pull 内 `Upsert` 设备游标后返回响应；**无客户端 ACK**。响应丢失时通常可重拉；若裁剪已抬 floor 则 `RequiresFullSync` | **暂不改协议**；版本探测仅 `TouchAsync` 不抬游标；裁剪与 floor 同事务；全量有抢救快照。ACK 协议变更单独立项 |
| 历史非全库 | Revision 不覆盖文件夹树 | 全库 Archive + local.db 备份 |
| 服务端备份 | 应用内不内置 Litestream | **已提供**运维脚本与文档：[`scripts/server-backup/`](../../scripts/server-backup/)、[服务端备份与恢复运维指南](./服务端备份与恢复运维指南.md) |
| integrity 失败 | 仅日志，未自动弹窗 | **启动弹窗引导打开备份管理** |
| 回收站硬删 | 到期仅删本机副本，云端仍为软删；未同步 Delete 会保留 pending | 云端软删为设计如此；永久删需另做服务端 purge |
