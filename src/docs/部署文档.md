# BiSheng Server 部署文档

## Ubuntu 22.04/24.04 + Apache 反向代理 + CertBot SSL

本文档描述如何将 BiSheng Server 部署到一台公网 Ubuntu 服务器上，使用 Apache 作为反向代理，CertBot 自动申请和续签 Let's Encrypt 免费证书。

### 架构概览

```
客户端 (HTTPS)
    │
    ▼
┌─────────────────────────────┐
│  Apache (443 / 80)          │
│  ├── SSL 终结 (CertBot)      │
│  ├── HTTP→HTTPS 重定向       │
│  ├── HSTS Header 注入        │
│  └── 反向代理 → 127.0.0.1:8090 │
└──────────────┬──────────────┘
               │ HTTP (本机回环)
               ▼
┌─────────────────────────────┐
│  BiSheng Server (Kestrel)   │
│  监听 127.0.0.1:8090        │
│  UseForwardedHeaders 解析   │
│  真实客户端 IP / 外部协议    │
│  ├── SQLite: bisheng.db     │
│  └── 文件: uploads/          │
└─────────────────────────────┘
```

Apache 通过 `X-Forwarded-For` / `X-Forwarded-Proto` 向 Kestrel 传递真实客户端信息，供登录限流与安全日志使用。设计细节见 [反向代理与真实IP传递设计文档.md](./反向代理与真实IP传递设计文档.md)。

## 3. 发布应用

### 3.1 在开发机构建

在开发机上执行：

```bash
cd src/BiSheng.Server
dotnet publish -c Release -o ./publish --self-contained false
```

产出目录 `publish/` 包含所有运行所需文件。

### 3.2 上传到服务器

```bash
# 在服务器上创建目录
sudo mkdir -p /opt/bisheng

# 从开发机上传（或 scp / rsync）
scp -r publish/* user@your-server:/opt/bisheng/
```

### 3.3 目录结构

```
/opt/bisheng/
├── BiSheng.Server.dll       # 主程序
├── BiSheng.Server.deps.json
├── BiSheng.Server.runtimeconfig.json

├── bisheng.db                    ← 首次运行自动创建
├── uploads/                      ← 图片上传目录（自动创建）
└── ...
```

```bash
sudo nano /opt/bisheng/appsettings.Production.json
```

## 4. 创建 Systemd 服务

Kestrel **仅监听本机回环**，由 Apache 对外暴露 80/443。必须设置 `ASPNETCORE_ENVIRONMENT=Production`，应用才会信任本机 Apache 传入的 `X-Forwarded-*` 头并解析真实客户端 IP。

```bash
sudo nano /etc/systemd/system/bisheng.service
```

```ini
[Unit]
Description=BiSheng Server - Note Sync Service
After=network.target

[Service]
Type=notify
User=bisheng
Group=bisheng
WorkingDirectory=/opt/bisheng
ExecStart=/usr/bin/dotnet /opt/bisheng/BiSheng.Server.dll

# ===== 环境变量（与 Apache ProxyPass 目标一致） =====
# Production：启用 ForwardedHeaders，仅信任 127.0.0.1 / ::1 反代
Environment=ASPNETCORE_ENVIRONMENT=Production
# 仅本机监听，禁止公网直连 Kestrel
Environment=ASPNETCORE_URLS=http://127.0.0.1:8090
Environment=DOTNET_PRINT_TELEMETRY_MESSAGE=false

# ===== 进程管理 =====
Restart=always
RestartSec=10
KillSignal=SIGINT
SyslogIdentifier=bisheng

# ===== 安全加固 =====
ProtectSystem=full
ProtectHome=true
NoNewPrivileges=true
PrivateTmp=true

[Install]
WantedBy=multi-user.target
```

| 变量 | 值 | 说明 |
|------|-----|------|
| `ASPNETCORE_ENVIRONMENT` | `Production` | 加载 `appsettings.Production.json`（默认 Kestrel 仅监听 `127.0.0.1:8090`）；启用 KnownProxy；管理员登录要求 TOTP |
| `ASPNETCORE_URLS` | `http://127.0.0.1:8090` | 与 Apache `ProxyPass` 目标一致；**覆盖**配置文件中的监听地址时请仍保持仅本机 |

> **Data Protection 密钥：** 默认写在应用内容根下的 `data-protection-keys/`（例如 `/opt/bisheng/data-protection-keys`）。可用配置项 `DataProtection:KeysPath` 覆盖。请保证服务账号可读写该目录，否则重启后管理端 Cookie / 两步登录会话会失效。

> **Cookie `Secure` 与 `X-Forwarded-Proto`（注意事项）：**
>
> - 浏览器只在 **HTTPS** 下发送带 `Secure` 标记的 Cookie。生产配置 `Cookies:SecureAlways=true`（见 `appsettings.Production.json`）时，**管理端主会话 Cookie** 与 **两步登录 / 初始化的 pending Cookie** 都会强制打上 `Secure`（与是否已解析出 HTTPS 无关）。
> - Apache → Kestrel 通常是本机 **HTTP**。若 HTTPS 站点的 VirtualHost **未**设置 `RequestHeader set X-Forwarded-Proto "https"`，或 `mod_headers` 未启用，Kestrel 可能一直认为请求不是 HTTPS。此时限流/日志的真实 IP、以及依赖「当前是否 HTTPS」的其它逻辑会异常；主会话与 pending 虽仍因 `SecureAlways` 带 Secure，但整体安全边界仍依赖反代正确传协议。
> - **必须：** 443 站点配置 `X-Forwarded-Proto "https"`，80 站点可为 `"http"`；并保持 `ASPNETCORE_ENVIRONMENT=Production` 以便信任本机反代的 Forwarded 头。校验命令见下文 [§10 故障排查](#10-故障排查) 与 [反向代理与真实IP传递设计文档](./反向代理与真实IP传递设计文档.md)。

启用并启动：

```bash
sudo systemctl daemon-reload
sudo systemctl enable bisheng        # 开机自启
sudo systemctl start bisheng         # 立即启动
sudo systemctl status bisheng        # 查看状态
```

验证应用运行：

```bash
# 确认仅本机监听 8090
ss -tlnp | grep 8090

curl http://127.0.0.1:8090/admin/login
# 应返回 HTML 内容（302 跳转到 setup 或登录页）
```

## 5. 安装配置 Apache

Apache 负责 SSL、反向代理，并通过 **Forwarded Headers** 把真实客户端 IP 与外部协议传给 Kestrel。完整原理见 [反向代理与真实IP传递设计文档.md](./反向代理与真实IP传递设计文档.md)。

### 5.1 安装 Apache

```bash
sudo apt-get update
sudo apt-get install -y apache2
```

### 5.2 启用必要模块

```bash
sudo a2enmod proxy
sudo a2enmod proxy_http
sudo a2enmod proxy_wstunnel     # SignalR WebSocket 支持
sudo a2enmod rewrite
sudo a2enmod headers
sudo a2enmod ssl

sudo systemctl restart apache2
```

> **`headers` 模块必须启用**，否则下文 `RequestHeader set X-Forwarded-*` 不会生效，Kestrel 看到的 IP 将始终是 `127.0.0.1`。

### 5.3 创建 VirtualHost 配置

先创建 HTTP 版本（CertBot 需要）：

```bash
sudo nano /etc/apache2/sites-available/bisheng.conf
```

```apache
<VirtualHost *:80>
    ServerName notes.example.com

    # CertBot 验证路径
    DocumentRoot /var/www/html

    # === SignalR WebSocket ===
    RewriteEngine On
    RewriteCond %{HTTP:Upgrade} =websocket [NC]
    RewriteRule /hubs/sync(.*) ws://127.0.0.1:8090/hubs/sync$1 [P,L]

    # === 反向代理 → Kestrel（与 systemd ASPNETCORE_URLS 一致） ===
    ProxyPreserveHost On
    ProxyPass / http://127.0.0.1:8090/
    ProxyPassReverse / http://127.0.0.1:8090/

    # === 传递真实客户端信息（Kestrel UseForwardedHeaders 依赖） ===
    RequestHeader set X-Forwarded-Proto "http"
    RequestHeader set X-Forwarded-For "%{REMOTE_ADDR}s"

    ErrorLog ${APACHE_LOG_DIR}/bisheng-error.log
    CustomLog ${APACHE_LOG_DIR}/bisheng-access.log combined
</VirtualHost>
```

启用站点：

```bash
sudo a2ensite bisheng
sudo systemctl reload apache2
```

此时访问 `http://notes.example.com` 应该能看到 BiSheng 的初始化页面。

## 6. CertBot 申请 SSL 证书

### 6.1 安装 CertBot

```bash
sudo apt-get install -y certbot python3-certbot-apache
```

### 6.2 申请证书

```bash
sudo certbot --apache -d notes.example.com
```

CertBot 会自动：
1. 验证域名所有权（通过 HTTP 挑战）
2. 申请 Let's Encrypt 证书
3. 修改 Apache 配置添加 SSL 部分
4. 设置自动续签定时任务

按提示操作：
- 输入邮箱（用于证书过期提醒）
- 同意服务条款
- 选择是否重定向 HTTP → HTTPS（选择 2 - Redirect，推荐）

### 6.3 验证自动续签

```bash
# 测试续签（不会实际续签）
sudo certbot renew --dry-run
```

CertBot 已自动创建 `/etc/cron.d/certbot` 定时任务，每天检查两次。

### 6.4 完善 SSL 配置

CertBot 修改后的配置可能缺少安全参数，手动优化：

```bash
sudo nano /etc/apache2/sites-available/bisheng-le-ssl.conf
```

在 `<VirtualHost *:443>` 块内添加/修改：

```apache
<VirtualHost *:443>
    ServerName notes.example.com

    # === SSL 配置（CertBot 已生成，以下为安全加固） ===
    SSLCertificateFile /etc/letsencrypt/live/notes.example.com/fullchain.pem
    SSLCertificateKeyFile /etc/letsencrypt/live/notes.example.com/privkey.pem
    Include /etc/letsencrypt/options-ssl-apache.conf

    # === 安全 Header ===
    Header always set Strict-Transport-Security "max-age=31536000; includeSubDomains"
    Header always set X-Content-Type-Options "nosniff"
    Header always set X-Frame-Options "DENY"
    Header always set Referrer-Policy "strict-origin-when-cross-origin"

    # === 传递真实客户端信息（Kestrel UseForwardedHeaders 依赖） ===
    RequestHeader set X-Forwarded-Proto "https"
    RequestHeader set X-Forwarded-For "%{REMOTE_ADDR}s"

    # === SignalR WebSocket 代理 ===
    RewriteEngine On
    RewriteCond %{HTTP:Upgrade} =websocket [NC]
    RewriteRule /hubs/sync(.*) ws://127.0.0.1:8090/hubs/sync$1 [P,L]

    # === 反向代理 → Kestrel（与 systemd ASPNETCORE_URLS 一致） ===
    ProxyPreserveHost On
    ProxyPass / http://127.0.0.1:8090/
    ProxyPassReverse / http://127.0.0.1:8090/

    # === 上传文件大小限制（按需调整） ===
    LimitRequestBody 52428800

    # === 日志 ===
    ErrorLog ${APACHE_LOG_DIR}/bisheng-ssl-error.log
    CustomLog ${APACHE_LOG_DIR}/bisheng-ssl-access.log combined
</VirtualHost>
```

```bash
sudo apache2ctl configtest   # 检查语法
sudo systemctl reload apache2
```

### 6.5 验证真实 IP 传递

完成 HTTP / HTTPS VirtualHost 配置并重载 Apache 后，确认 Kestrel 能拿到公网客户端 IP：

```bash
# Apache 配置中已有 Forwarded 头
grep -E 'X-Forwarded-(For|Proto)' /etc/apache2/sites-available/bisheng*.conf

# mod_headers 已加载
apache2ctl -M | grep headers

# systemd 环境为 Production
sudo systemctl show bisheng -p Environment

# 经 HTTPS 故意登录失败，journal 中 IP 应为公网地址
sudo journalctl -u bisheng -f
```

若日志中 IP 始终为 `127.0.0.1`，参见 [故障排查 § Rate Limiter / 真实 IP 异常](#rate-limiter--真实-ip-异常)。

```bash
# 安装（一次性）
```

脚本会：升级前备份 → 停服 → 校验哈希 → 替换程序文件（保留 db / uploads / Production 配置 / DP keys）→ 启服与健康检查。说明见 [`scripts/server-upgrade/README.md`](../../scripts/server-upgrade/README.md)。

手工备选步骤：

```bash
# 1. 停止服务
sudo systemctl stop bisheng

# 2. 备份数据库、uploads 与配置（推荐脚本；勿对运行中的库直接 cp）

# 若尚未安装脚本，见 §9.3 / 服务端备份与恢复运维指南

# 3. 上传新版本文件（保留 bisheng.db、appsettings.Production.json、uploads/、data-protection-keys/）
```

### 9.3 备份策略

应用**不内置** Litestream。生产环境必须由运维备份 **`bisheng.db` + `uploads/`**（建议含 `appsettings.Production.json` 与 `data-protection-keys/`）。

完整说明与恢复演练：[服务端备份与恢复运维指南](./服务端备份与恢复运维指南.md)。脚本目录：仓库 [`scripts/server-backup/`](../../scripts/server-backup/)。

## 10. 故障排查

### 502 Bad Gateway

应用未启动或端口不匹配：

```bash
sudo systemctl status bisheng
curl http://127.0.0.1:8090/admin/login    # 直接测试 Kestrel
```

### HTTPS 重定向循环

应用层已移除 `UseHttpsRedirection()`，HTTPS 强制完全由 Apache 负责。如仍出现循环，检查 Apache 配置：

```bash
# 确认 80 端口 VirtualHost 有重定向配置
grep -A2 'Redirect\|Rewrite' /etc/apache2/sites-available/bisheng.conf
# 确认 443 端口 VirtualHost 正确传递 X-Forwarded-Proto
grep X-Forwarded-Proto /etc/apache2/sites-available/bisheng-le-ssl.conf
```

### SignalR 连接失败

检查 WebSocket 模块是否启用：

```bash
apache2ctl -M | grep proxy_wstunnel
# 如无输出：sudo a2enmod proxy_wstunnel && sudo systemctl restart apache2
```

### Rate Limiter / 真实 IP 异常

限流**仅作用于** POST `/admin/login`（管理后台登录）。**笔记同步（`/api/sync`）、SignalR、其它 API 不会被 429。** 详见 [认证与安全设计文档 §3.2](./认证与安全设计文档.md#32-限流范围)。

所有请求在 Kestrel 侧显示为 `127.0.0.1` 时，登录限流会对所有用户共用一个配额，日志中的 IP 也无法用于排查。

**逐项检查：**

```bash
# 1. Apache 是否写入 Forwarded 头（HTTP 与 HTTPS 站点均需配置）
grep -E 'X-Forwarded-(For|Proto)' /etc/apache2/sites-available/bisheng*.conf

# 2. mod_headers 是否启用
apache2ctl -M | grep headers
# 无输出：sudo a2enmod headers && sudo systemctl restart apache2

# 3. Kestrel 是否为 Production（否则不解析 Forwarded 头）
sudo systemctl show bisheng -p Environment | grep Production

# 4. Kestrel 是否仅本机监听（与 ProxyPass 一致）
ss -tlnp | grep 8090
```

**常见原因：**

| 原因 | 处理 |
|------|------|
| 未配置 `RequestHeader set X-Forwarded-For` | 按 [§5.3 / §6.4](#53-创建-virtualhost-配置) 补全 Apache 配置 |
| `mod_headers` 未启用 | `sudo a2enmod headers` |
| 未设置 `ASPNETCORE_ENVIRONMENT=Production` | 修改 [§4 Systemd](#4-创建-systemd-服务) 并 `daemon-reload` |
| 公网直连 Kestrel 测试 | 限流/日志应经 Apache 访问后再验证 |
| HTTPS 站点缺少 `X-Forwarded-Proto: https` | 按 §6.4 补全；否则 Kestrel 不知对外协议，真实 IP/协议相关行为异常。生产下主会话与 pending Cookie 虽由 `Cookies:SecureAlways` 强制 Secure，仍须正确传递 Proto |

### 管理后台 Cookie / 两步登录异常

症状包括：登录后 Cookie 未带上、两步验证（TOTP）中断、浏览器提示 Cookie 被拒绝等。

**注意：**

1. 生产务必 `Cookies:SecureAlways=true`（`appsettings.Production.json` 默认已开）：主会话与 pending Cookie **一律** `Secure`，只在 HTTPS 下由浏览器发送。
2. 对外必须经 Apache **HTTPS** 访问管理面板；不要用 `http://公网IP:8090` 直连 Kestrel 做正式登录。
3. 443 VirtualHost 必须设置 `X-Forwarded-Proto "https"`，以便应用识别外部协议（与限流真实 IP 同源配置）。缺省时即使 Cookie 已 Secure，协议感知仍可能错误。

```bash
# 确认 Production 已加载 SecureAlways（配置文件）
grep -A2 SecureAlways /opt/bisheng/appsettings.Production.json
```
