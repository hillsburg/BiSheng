# BiSheng Server 日志设计文档

## 概述

BiSheng Server 使用 ASP.NET Core 内置的 `ILogger` 抽象，默认输出到**控制台**（stdout）。在 Linux 部署时，日志由 **systemd journal** 收集，可通过 `journalctl` 查看。

设计原则：

- **生产环境**：默认级别 `Warning`，压低框架噪音；业务命名空间 `BiSheng.Server` 保留 `Information`
- **开发环境**：默认级别 `Information`，未处理异常记录后仍抛出，保留开发者异常页
- **关键路径**：认证失败、同步 Push 失败、未处理异常必须可追溯
- **安全**：日志中不记录 API Key、密码、TOTP 验证码等敏感信息

与客户端日志（`BiSheng.Latte` 的 NLog + 文件落盘）相互独立，参见 [LogHelper使用指南.md](./LogHelper使用指南.md)。

## 配置文件

| 文件 | 生效环境 | 说明 |
|------|----------|------|
| `appsettings.json` | 所有环境（基础） | `Default: Information` |
| `appsettings.Development.json` | `ASPNETCORE_ENVIRONMENT=Development` | 与基础配置相同 |
| `appsettings.Production.json` | `ASPNETCORE_ENVIRONMENT=Production` | 生产分级，见下文 |

部署时必须设置：

```bash
Environment=ASPNETCORE_ENVIRONMENT=Production
```

（systemd 服务文件中已配置，见 [部署文档.md](./部署文档.md)）

### 生产环境级别（`appsettings.Production.json`）

```json
{
  "Logging": {
    "LogLevel": {
      "Default": "Warning",
      "Microsoft.Hosting.Lifetime": "Information",
      "BiSheng.Server": "Information",
      "Microsoft.EntityFrameworkCore": "Warning",
      "Microsoft.AspNetCore.SignalR": "Warning"
    }
  }
}
```

| 命名空间 | 级别 | 原因 |
|----------|------|------|
| `Default` | Warning | 过滤 ASP.NET Core 框架中大量 Information 日志 |
| `Microsoft.Hosting.Lifetime` | Information | 保留进程启动、停止、监听地址 |
| `BiSheng.Server` | Information | 保留 GC、图片上传等业务 Info |
| `Microsoft.EntityFrameworkCore` | Warning | 避免 SQL 语句刷屏 |
| `Microsoft.AspNetCore.SignalR` | Warning | 避免连接/心跳 Info 刷屏 |

### 开发 / 默认环境

```json
{
  "Logging": {
    "LogLevel": {
      "Default": "Information",
      "Microsoft.AspNetCore": "Warning"
    }
  }
}
```

## 日志级别约定

| 级别 | 用途 | 示例 |
|------|------|------|
| **Information** | 正常业务事件，频率低或可接受 | GC 完成、图片上传、服务启动 |
| **Warning** | 可恢复异常、认证失败、部分业务失败 | API Key 无效、登录失败、Push 单条变更跳过 |
| **Error** | 需人工关注的失败 | Push 事务回滚、GC 整体异常、未处理异常 |

当前代码**未使用** `LogDebug` / `LogTrace`。若后续增加调试日志，建议仅在 `Development` 环境启用，或通过配置将 `BiSheng.Server` 临时调到 `Debug`。

## 关键路径日志

### 1. API Key 认证失败

**位置**：`Auth/ApiKeyAuthHandler.cs`  
**级别**：Warning  
**触发**：请求携带 Key，但哈希不匹配或 Key 已停用  

```
API Key 认证失败: {RemoteIp} {Method} {Path}
```

不记录 Key 原文或哈希值。

### 2. 管理后台登录失败

**位置**：`Pages/Login.cshtml.cs`  
**级别**：Warning  

| 场景 | 消息模板 |
|------|----------|
| 用户名或密码错误 | `管理后台登录失败: 用户名={Username}, IP={RemoteIp}` |
| TOTP 验证码错误 | `管理后台 TOTP 验证失败: 用户名={Username}, IP={RemoteIp}` |

不记录密码或 TOTP 码。

### 2.1 登录 POST 限流（429）

**位置**：`Program.cs`（`AddRateLimiter` → `OnRejected`）  
**级别**：Warning  
**触发**：同一 IP 在 1 分钟内登录 POST 超过 5 次  

```
请求被限流: {Method} {Path}, IP={RemoteIp}
```

Logger 类别：`BiSheng.Server.RateLimiting`。GET 加载登录页不计入限流。

**不受此限流影响**：`/api/sync`、SignalR `/hubs/sync` 及所有 API Key 接口（详见 [认证与安全设计文档 §3.2](./认证与安全设计文档.md#32-限流范围)）。

### 3. 同步 Push

**位置**：`Services/SyncService.cs`

| 场景 | 级别 | 消息 |
|------|------|------|
| 单条变更校验失败 / 实体不存在 | Warning | `跳过无效变更 {EntityType}:{EntityId}, 用户: {UserId}` |
| 单条变更应用抛异常 | Warning | `应用变更失败 {EntityType}:{EntityId}, 用户: {UserId}`（含异常栈） |
| 事务提交 / SaveChanges 失败 | **Error** | `同步 Push 事务失败, 用户: {UserId}, 变更数: {Count}`（含异常栈） |
| 事务成功但部分变更失败 | Warning | `同步 Push 部分失败, 用户: {UserId}, 失败: {Failed}, 成功: {Applied}` |

Pull、version 查询等读操作**不记录**日志（避免同步频繁时刷屏）。

### 4. 未处理异常

按环境分两种处理方式（`Program.cs`）：

| 环境 | 机制 | 行为 |
|------|------|------|
| Development | `ExceptionLoggingMiddleware` | 记录 Error 后**重新抛出**，保留开发者异常页 |
| Production | `UseExceptionHandler` | 记录 Error 后返回 **HTTP 500**，响应体为 `Internal Server Error` |

日志消息：

```
未处理异常 {Method} {Path}
```

Logger 类别：`BiSheng.Server.UnhandledException`（生产）或 `ExceptionLoggingMiddleware`（开发）。

### 5. 图片与后台任务

**位置**：`Controllers/ImagesController.cs`、`Services/ImageGarbageCollector.cs`、`Services/Images/ImageCleanupService.cs`

| 场景 | 级别 |
|------|------|
| 图片上传成功 | Information |
| GC 服务启动 | Information |
| GC 清理完成（有删除时） | Information |
| 孤儿 soft-delete / dry-run 计数 | Information |
| 单个文件删除失败 | Warning |
| GC 扫描整体异常 | Error |

## 异常处理管道顺序

```
app.Build()
    │
    ├── Development → ExceptionLoggingMiddleware（记录 + 重抛）
    │
    └── Production  → UseExceptionHandler（记录 + 500 响应）
    │
    ▼
UseForwardedHeaders → RateLimiter → CORS → SetupMiddleware → Auth → 路由
```

`UseExceptionHandler` / `ExceptionLoggingMiddleware` 位于管道最前，可捕获后续中间件与 Controller 中的未处理异常。

## 运维：查看日志

### systemd（推荐）

```bash
sudo journalctl -u bisheng -f              # 实时跟踪
sudo journalctl -u bisheng --since today   # 今日
sudo journalctl -u bisheng -p err          # 仅 Error 及以上
```

### 过滤示例

```bash
# 认证失败
sudo journalctl -u bisheng | grep "认证失败"

# 登录限流
sudo journalctl -u bisheng | grep "请求被限流"

# Push 事务失败
sudo journalctl -u bisheng | grep "Push 事务失败"

# 未处理异常
sudo journalctl -u bisheng | grep "未处理异常"
```

### 当前限制

- 日志**仅输出到控制台**，无内置文件轮转
- 长期保留依赖 journald 配置或外部日志采集（如 Loki、ELK）
- 未启用 ASP.NET Core `HttpLogging`，**不记录**每个 HTTP 请求的访问日志

Apache 访问日志与 Kestrel 真实 IP 传递见 [反向代理与真实IP传递设计文档.md](./反向代理与真实IP传递设计文档.md)。

## 与客户端日志的对比

| 项目 | BiSheng Server | BiSheng.Latte |
|------|----------------|---------------|
| 实现 | `ILogger` + 控制台 | NLog + 本地文件 |
| 配置 | `appsettings.*.json` | `NLog.config` |
| 落盘 | 依赖 journald / 容器日志 | `log/app-*.log`、`log/error-*.log` |
| 文档 | 本文档 | [LogHelper使用指南.md](./LogHelper使用指南.md) |

## 扩展建议

若后续需要增强，可按优先级考虑：

1. **Serilog + 滚动文件**：自托管单机部署时便于离线排查
2. **Push Pull 采样日志**：仅在 Debug 或配置开关开启时记录
3. **结构化日志**：统一使用 `{Named}` 占位符（当前已遵循），便于 journald / 日志平台检索
