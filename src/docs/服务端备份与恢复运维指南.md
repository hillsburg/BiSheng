# 服务端备份与恢复运维指南

本文描述 **BiSheng Server** 侧 `bisheng.db` + `uploads/` 的推荐备份与恢复流程。应用**不内置** Litestream；连续复制由运维可选接入。

相关脚本：[`scripts/server-backup/`](../../scripts/server-backup/)。客户端本地 `local.db` 备份见 [数据安全策略设计文档](./数据安全策略设计文档.md) 与 [同步与安全设置使用指南](./同步与安全设置使用指南.md)。

---

## 1. 要备份什么

| 路径 | 必要性 | 说明 |
|------|--------|------|
| `bisheng.db`（及恢复时勿混用旧 WAL） | **必须** | 笔记、文件夹、SyncLog、用户与图片元数据 |
| `uploads/` | **必须** | 图片二进制；仅备库会导致「元数据在、文件丢」 |
| `appsettings.Production.json` | 建议 | 生产绑定、反代相关配置 |
| `data-protection-keys/` | 建议 | Cookie/DP 密钥；丢失会导致已有加密 Cookie 失效 |

默认部署根目录：`/opt/bisheng`（可用环境变量 `BISHENG_HOME` 覆盖）。

---

## 2. 推荐：在线一致快照（日常）

**不要**对正在运行的 SQLite 做简单 `cp bisheng.db`：WAL 模式下可能得到不一致副本。

使用仓库脚本（内部调用 `sqlite3 … ".backup"`）：

```bash
# 一次性安装
sudo apt install -y sqlite3
sudo mkdir -p /opt/bisheng/backup
sudo cp /path/to/repo/scripts/server-backup/backup-bisheng.sh /usr/local/bin/
sudo chmod +x /usr/local/bin/backup-bisheng.sh

# 手动跑一轮
sudo BISHENG_HOME=/opt/bisheng BACKUP_ROOT=/opt/bisheng/backup RETENTION_DAYS=14 \
  /usr/local/bin/backup-bisheng.sh
```

成功后目录形如：

```text
/opt/bisheng/backup/20260716-031500/
  bisheng.db
  uploads.tar.gz
  appsettings.Production.json   # 若存在
  data-protection-keys.tar.gz   # 若存在
  manifest.txt
```

定时任务示例见 [`scripts/server-backup/crontab.example`](../../scripts/server-backup/crontab.example)。

### 环境变量

| 变量 | 默认 | 说明 |
|------|------|------|
| `BISHENG_HOME` | `/opt/bisheng` | 安装根目录 |
| `BACKUP_ROOT` | `$BISHENG_HOME/backup` | 备份输出根 |
| `DB_PATH` | `$BISHENG_HOME/bisheng.db` | 数据库路径 |
| `UPLOADS_DIR` | `$BISHENG_HOME/uploads` | 图片目录 |
| `RETENTION_DAYS` | `14` | 删除早于该天数的备份子目录；`0` 表示不清理 |

---

## 3. 升级前冷备份（可选）

重大升级或迁移前可停服后再拷贝，语义更简单：

```bash
sudo systemctl stop bisheng
sudo BISHENG_HOME=/opt/bisheng /usr/local/bin/backup-bisheng.sh
# 或在停服后对主库使用 sqlite3 ".backup" / 安全复制
sudo systemctl start bisheng
```

---

## 4. 可选：Litestream 连续复制

适用：需要比日备更细的时间点恢复。配置示例：[`scripts/server-backup/litestream.yml.example`](../../scripts/server-backup/litestream.yml.example)。

注意：

1. Litestream **只复制 SQLite**；`uploads/` 仍须 `backup-bisheng.sh`、rsync 或 restic 等另行备份。
2. 应用进程**不会**自动安装或启动 Litestream。
3. 日备脚本与 Litestream 可并存：前者做可携带的整包快照，后者做连续 WAL 复制。

恢复（示意，以官方文档为准）：

```bash
sudo systemctl stop bisheng
sudo litestream restore -o /opt/bisheng/bisheng.db /var/backups/bisheng-litestream
# 再按同时间点的 uploads 备份恢复 uploads/
sudo systemctl start bisheng
```

---

## 5. 恢复演练清单

至少每季度做一次**非生产或临时目录**演练：

1. 选取一份 `BACKUP_ROOT/<stamp>/` 备份。
2. `sudo systemctl stop bisheng`。
3. 执行：

   ```bash
   sudo BISHENG_HOME=/opt/bisheng \
     /path/to/repo/scripts/server-backup/restore-bisheng.sh \
     /opt/bisheng/backup/<stamp>
   ```

4. `sudo systemctl start bisheng`，检查：
   - `curl -sS -o /dev/null -w '%{http_code}\n' http://127.0.0.1:8090/admin/login`（期望非 000）
   - 管理面板可登录；任选一篇含图笔记，图片可加载
5. 脚本会把覆盖前的库移到 `pre-restore-*`；确认无误后再删。

---

## 6. 与客户端备份的关系

| 层级 | 保护对象 | 不能替代 |
|------|----------|----------|
| 客户端 L1 自动备份 | 本机 `local.db` | 不能恢复服务器为真相源时的多端数据 |
| 服务端 L3（本文） | `bisheng.db` + `uploads/` | 不能恢复从未 Push 的本机编辑 |
| 全库 Archive 导出 | 用户主动异地副本 | 不替代定时运维备份 |
