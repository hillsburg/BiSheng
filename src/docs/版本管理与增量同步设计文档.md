# 版本管理与增量同步设计文档

## 一、概述

BiSheng 的同步系统采用**基于版本号的增量同步**策略：每次数据变更分配一个全局单调递增的版本号，客户端通过记录"上次同步到的版本号"来拉取增量变更，避免全量传输。

### 核心设计目标

| 目标 | 实现方式 |
|------|----------|
| 增量传输 | 客户端只拉取 `Version > LastSyncVersion` 的变更 |
| 变更有序 | 版本号单调递增，保证变更按时间顺序应用 |
| 冲突检测 | 通过版本号判断客户端是否落后于服务端 |
| 多设备一致 | 所有设备最终收敛到相同的版本号和数据状态 |

## 二、版本号定义

### 2.1 存储位置

版本号以 `long`（Int64）类型存储在多个位置：

| 表/实体 | 字段 | 说明 |
|---------|------|------|
| `SyncLog`（服务端） | `Version` | 每条变更记录的版本号，变更审计源 |
| `UserSyncMeta`（服务端） | `CurrentVersion` | **版本计数器**：当前已分配的最高版本号（权威读源） |
| `Note` / `Folder`（服务端） | `Version` | 实体当前版本号（冗余，便于查询） |
| `LocalNote` / `LocalFolder`（客户端） | `Version` | 本地实体的版本号副本 |
| `LocalSyncState`（客户端） | `LastSyncVersion` | 客户端最后成功同步的服务端版本号 |

### 2.2 作用域

版本号是**每用户独立**的。不同用户各自从 1 开始递增，互不影响：

```
用户 A 的版本序列：1, 2, 3, 4, 5, ...
用户 B 的版本序列：1, 2, 3, ...（独立计数）
```

### 2.3 数值范围

`long` 最大值为 `9,223,372,036,854,775,807`（约 9.2 × 10¹⁸）。

以每天 1,000 次变更估算，需要约 **25 亿年**才能耗尽，实际上不存在溢出风险。

## 三、版本号分配

### 3.1 设计原则

版本号由 `UserSyncMeta.CurrentVersion` **原子递增**分配，不再使用 `MAX(SyncLogs.Version) + 1`：

| 旧方案 | 问题 | 新方案 |
|--------|------|--------|
| 每条变更前 `SELECT MAX(Version)` | 并发 Push / REST 可能拿到相同 Max，触发唯一索引冲突 | SQLite `UPDATE … RETURNING` 单语句递增 |
| `GET /api/sync/version` 扫描 SyncLog | 日志量大时额外开销 | 直接读 `UserSyncMeta.CurrentVersion` |

实现：`UserSyncVersionService`（`BiSheng.Server/Services/UserSyncVersionService.cs`）。

### 3.2 服务端 REST 与 Push 统一写路径（P0-A）

REST 与 Sync Push **共用** `IEntityChangeWriter`：校验通过后才 `ReserveNextVersionAsync`，再写入实体、SyncLog、NoteRevision（如有）。Writer **不负责** `SaveChanges`、事务 Commit 或 SignalR。

```
NotesController / FoldersController / NoteRevisionsController
  → INoteMutationService / IFolderMutationService（开事务 → Writer → Save → Commit → Notify）

SyncController → ISyncService.PushAsync（单事务内循环 Writer → Commit → NotifyBatch）
```

适用控制器：`NotesController`、`FoldersController`、`NoteRevisionsController`（恢复历史委托 `INoteMutationService.RestoreFromRevisionAsync`）。

> 架构细节与 PR 拆分见 [P0统一写路径与客户端单测设计文档](./P0统一写路径与客户端单测设计文档.md)。

### 3.3 客户端 Push 批量推送

Push 在**事务内**逐条调用 `IEntityChangeWriter.TryApplyAsync`；每条变更**校验通过之后**才分配版本号，避免无效变更消耗序号：

```csharp
foreach (var change in orderedChanges)
{
    var result = await _entityChangeWriter.TryApplyAsync(
        _db, userId, change.ToEntityMutation(), batchContext, options, ct);

    if (result is MutationSkipped)
        failedEntityIds.Add(change.EntityId);
    else if (result is MutationApplied applied)
        appliedList.Add(applied.Applied);
}

await _db.SaveChangesAsync(ct);
await tx.CommitAsync(ct);
await _notifier.NotifyBatchAsync(userId, appliedList, ct);
```

> **演进说明**：早期版本在批次开头预计算 `nextVersion` 并手动递增；更早版本在每条变更内部查询 `Max(Version)`，同一批次可能重复。P0 起 REST/Push 共用 Writer；版本仍为 per-change 原子分配 + 事务内 `ClientSyncState` 一并提交，避免 `database is locked`。

### 3.4 首次使用与迁移对齐

- 新用户首次分配版本时，`EnsureInitializedAsync` 创建 `UserSyncMeta`，`CurrentVersion` 从现有 `SyncLogs` 的 `MAX(Version)` 回填（兼容旧库）。
- 迁移 `AddUserSyncCurrentVersion` 对已有 `UserSyncMetas` 行执行相同回填 SQL。

## 四、服务端同步架构

### 4.1 分层

```
HTTP 层          SyncController、NotesController、FoldersController、NoteRevisionsController
                    ↓
编排层           ISyncService（Pull / Push）
                 INoteMutationService / IFolderMutationService（REST）
                    ↓
写内核           IEntityChangeWriter（校验 + ReserveVersion + ChangeTracker）
                    ↓
Commit 后        ISyncChangeNotifier → SignalR
                    ↓
基础设施         UserSyncVersionService、ClientSyncStateService、NoteRevisionService
                    ↓
数据层           AppDbContext（SQLite + WAL）
```

Controller 与 `SyncService` 均不包含独立写入逻辑；Push 已删除 `ApplyClientChange`，改由 `EntityChangeWriter` 统一实现。

### 4.2 依赖注入（Microsoft.Extensions.Hosting）

服务注册集中在 `DependencyInjection/ServiceCollectionExtensions.cs`：

| 扩展方法 | 注册内容 |
|----------|----------|
| `AddBiShengServerCore()` | DbContext、认证、Swagger、`ISyncService`、`IEntityChangeWriter`、`ISyncChangeNotifier`、`INoteMutationService`、`IFolderMutationService`（Scoped）、`UserSyncVersionService` 等单例 |
| `AddBiShengBackgroundWorkers(configuration)` | `ImageCleanupService` + `ImageGarbageCollector`、`SyncLogCompactionService`；绑定 `ImageGc` |

`Program.cs` 调用上述扩展；CORS、RateLimiter、中间件管道仍保留在 `Program.cs`。

## 五、版本感知的同步流程

### 5.1 Push + Pull 流程

```
客户端                                    服务端
  │                                         │
  │  Push(clientVersion, pendingChanges)    │
  │────────────────────────────────────────→│
  │                                         │  1. 应用变更，分配新版本号
  │                                         │  2. 检查 clientVersion < serverVersion?
  │  PushResponse(serverVersion, conflicts) │     → 是：返回需要拉取的变更（终态折叠）
  │←────────────────────────────────────────│
  │                                         │
  │  if localVersion < serverVersion:       │
  │  loop until HasMore=false:              │
  │    Pull(since, limit=200)               │
  │────────────────────────────────────────→│
  │                                         │  本批终态折叠；返回
  │    PullResponse(changes, nextSince,     │  NextSince / HasMore /
  │                 hasMore, serverVersion) │  ServerVersion
  │←────────────────────────────────────────│
  │    cursor = nextSince                   │
  │  最终 LastSyncVersion = ServerVersion   │
  │                                         │
```

### 5.2 终态折叠法则

- 查询 `Version > since` 的 SyncLog（可选 `limit` 分页，按 Version 升序取本批）
- 对本批按 `EntityId` 分组去重，取每组最新的 Action
- 从实体表（Notes / Folders）查询该实体的**最新最终状态**
- 每个实体在本批只返回 1 条 `ChangeDto`（包含完整 Payload）
- 响应携带：
  - `ServerVersion`：服务端 tip（当前水位）
  - `NextSince`：本批消费到的最高 SyncLog 版本（客户端下次 `since`）
  - `HasMore`：是否还有后续批次

| 场景 | 返回结果 |
|------|----------|
| 笔记被编辑 50 次（同批内） | 1 条 `Update`（最新内容） |
| 文件夹创建后重命名 3 次（同批内） | 1 条 `Update`（最新名称） |
| 笔记创建后立即删除（同批内） | 1 条 `Delete` |
| `limit=200` 且落后 1000 条 | 多轮 Pull，每轮最多读 200 条 SyncLog |

**分页约定**（PR4）：

```
GET /api/sync/pull?since={n}&limit={pageSize}
```

- 默认 `limit=200`，上限 500；`limit≤0` 时亦按默认页大小分页（不再提供一次拉完模式）
- 客户端循环直到 `HasMore=false`
- 有后续批次时**服务端设备游标**推进到 `NextSince`；最后一批推进到 tip（`ServerVersion`）——推进发生在响应返回前，**无客户端 ACK**
- 中间批次：客户端本地游标 = `NextSince`；末批 = `ServerVersion`（客户端在本地 merge 成功后写入）
- `GET /api/sync/version` 仅刷新 `LastSeenAt`，不抬 `LastSyncVersion`

客户端的 `ApplyRemoteChangeToLocalDb` 采用 **upsert 语义**：`Update` 操作在本地不存在时自动创建，存在则覆盖。SignalR 仅推送元数据通知，客户端经 HTTP Pull 获取带 Payload 的变更后再应用。

> **设计说明：** 图片管道在下载失败时不推进图片游标；笔记 SyncLog Pull 目前无对等 ACK。响应丢失 + 裁剪竞态的后果是 `RequiresFullSync`，见 [SyncLog裁剪设计文档](./SyncLog裁剪设计文档.md) §5.1 与 [数据安全策略设计文档](./数据安全策略设计文档.md) §六。协议级 ACK 暂不实施。

### 5.3 版本感知策略

客户端通过三个阶段判断是否需要 Pull：

| 阶段 | 条件 | 行为 |
|------|------|------|
| Push 有待推送 | Push 返回 `ServerVersion` | 直接从响应获取，无需额外请求 |
| Push 无待推送 | 调用 `GET /api/sync/version` | 轻量级接口，仅返回当前版本号 |
| 版本比较 | `LastSyncVersion >= ServerVersion` | 跳过 Pull（已同步到最新） |
| 版本比较 | `LastSyncVersion < ServerVersion` | 执行 Pull 拉取增量变更 |

### 5.4 冲突检测

Push 响应中，如果 `clientVersion < serverVersion`，说明在客户端离线期间服务端已有新变更。服务端返回 `ConflictingChanges`（终态折叠后的变更），客户端据此进行冲突检测：

```
本地 PendingChanges 中有同一实体的变更 → 记录到 SyncConflict 表
本地无该实体的待推送变更 → 安全应用到本地 DB
```

## 六、SyncLog 增长分析

### 6.1 当前策略

SyncLog 只增不删——每次变更插入一条记录，永久保留。

### 6.2 增长估算

| 使用强度 | 日均变更 | 1 年累计 | 5 年累计 |
|----------|----------|----------|----------|
| 轻度使用 | ~10 次 | ~3,650 条 | ~18,250 条 |
| 中度使用 | ~100 次 | ~36,500 条 | ~182,500 条 |
| 重度使用 | ~1,000 次 | ~365,000 条 | ~1,825,000 条 |

SQLite 单表百万行级别无性能压力，个人笔记应用通常远低于此量级。

### 6.3 未来优化方向

> **已实现**：详见 [SyncLog裁剪设计文档](./SyncLog裁剪设计文档.md)。

当 SyncLog 行数显著增长时，系统通过以下机制自动裁剪（摘要）：

#### 策略 1：基于安全裁剪线的日志压缩（已实施）

裁剪的唯一安全依据是**所有活跃客户端中最小的 `LastSyncVersion`**：

```
安全裁剪线 = min(活跃 ClientSyncState.LastSyncVersion)
可安全删除: Version < 安全裁剪线 的所有 SyncLog 记录
```

实现要点：
- 每个 API Key（设备）在 Push/Pull 后更新 `ClientSyncStates`
- 后台服务 `SyncLogCompactionService` 定期执行 DELETE
- `UserSyncMetas.LogRetentionFloor` 记录已裁剪上界

#### 策略 2：僵尸客户端处理（已实施）

超过 `SyncLogStaleClientDays`（默认 90 天）未同步的设备标记为 stale，不再拖住裁剪线。再次上线时若版本低于 `LogRetentionFloor`，响应 `RequiresFullSync`，客户端全量重建。

#### ~~策略 3：服务端合并连续变更（不推荐）~~

> **此方案存在严重缺陷，不建议在服务端实施。**
>
> 假设同一笔记在 1 分钟内保存 5 次，产生 Version 1~5。如果客户端 A 已经拉走了 Version 2（`LastSyncVersion = 2`），随后服务端将 1~5 合并为一条新日志（分配新 Version 6），那么：
> - 客户端 A 的 `LastSyncVersion = 2` 在服务端日志链中不存在了
> - 下次 Pull `where Version > 2` 会跳过合并后的记录，丢失变更
>
> 服务端日志一旦写入并被任何客户端拉取，就**不可变**（Immutable Log）。

#### 替代方案：客户端防抖（已实施）

从源头减少密集变更产生的日志量，而非事后合并：

- `LocalChangeTracker` 对同一实体的多次变更做**去重合并**（只保留最新状态）
- `SyncService` 的事件驱动采用 **2 秒防抖**（`EventDebounceMs = 2000`），用户停止编辑后才触发 Push
- 连续快速保存 5 次，实际只会产生 1 条 `LocalPendingChange`，推送后也只产生 1 条 SyncLog

这是比服务端合并更安全、更简单的方案，且已在当前架构中实现。

## 八、笔记历史版本（与同步版本分离）

用户可见的**每篇笔记历史**（最多 100 条）由独立的 `NoteRevisions` / `LocalNoteRevisions` 表承载，**不复用**本节的 `SyncLog` 或全局 `Version`。

| 维度 | 同步 `Version` | 笔记 `RevisionNumber` |
|------|----------------|------------------------|
| 作用域 | 每用户全局变更序号 | 每篇笔记内 1..N |
| 用途 | Push / Pull 增量同步 | 历史查看与恢复 |
| 保留 | SyncLog 按裁剪线删除 | 每笔记 FIFO，最多 100 条 |
| Pull 语义 | 终态折叠，丢弃中间态 | 保留每次有效快照 |

写入时机摘要：服务端在 **Push / REST 更新成功**后，由 `NoteRevisionSampling` 按「有意义改动 + 最短间隔（默认 10 分钟）」决定是否写入 `NoteRevisions`（**不改变** Notes / SyncLog）；客户端本地历史见分层采样。恢复历史会写回当前笔记并以 `ForceNoteRevision` **再产生一条新 revision**；笔记软删后历史保留，需用户手动清理。

> **已实现**：详见 [笔记历史版本设计文档](./笔记历史版本设计文档.md)。
