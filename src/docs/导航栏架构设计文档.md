## 一、要解决的问题

早期实现里，导航栏与同步引擎、编辑器绑在一起，典型症状包括：

| 症状 | 根因 |
|------|------|
| 编辑时光标跳动、周期性「刷新感」 | 同步完成后全量 `Refresh()` + 无条件重写 `NoteEditor.Text` |
| 编辑器偶发被清空 | 多份 `LocalNote` 缓存 + 同步竞态，DB 空内容覆盖 UI |
| 树模式选中笔记后编辑器被清空 | `SelectedFolder` 引用被 Refresh 替换，触发 `LoadNotes` 链式反应 |
| 同步与 UI 强耦合 | `SyncService.OnSyncComplete` 直接驱动 `MainViewModel` 刷新导航 |
| 性能浪费 | 每条 SignalR 变更都重建整棵 `FolderTree` / `NoteList` |

架构目标可以概括为三句话：

1. **SQLite 是唯一持久化真相**；UI 是投影，可增量更新。  
2. **同步层不知道 WPF**；展示层不知道 HTTP/SignalR。  
3. **编辑器是会话**，远端变更只在「未编辑 + 版本前进」时合并。

## 二、分层总览（现行 P5）

```
┌─────────────────────────────────────────────────────────────────┐
│  展示层（WPF ViewModel + View）                                   │
│  FolderTreeViewModel · NoteListViewModel · NavigationViewModel    │
│  EditorViewModel（会话：焦点 / 脏 / loadedVersion）               │
└────────────────────────────▲────────────────────────────────────┘
                             │ 唯一入口
┌────────────────────────────┴────────────────────────────────────┐
│  NavigationPresentationCoordinator                               │
│  · UI 线程调度 · 编辑/重命名 deferred · 按投影类型分发             │
└────────────────────────────▲────────────────────────────────────┘
                             │ Changed
┌────────────────────────────┴────────────────────────────────────┐
│  INavigationReadModel（读模型边界）                               │
│  Publish(NavigationProjectionUpdate)                             │
└───────▲───────────────────────────────▲─────────────────────────┘
        │                               │
  SyncService                    INavigationMutationPublisher
  （远端 Pull/Push/SignalR）      （本地 CRUD · 搜索 · 布局）
        │                               │
        └───────────▶ SQLite ◀──────────┘
```

### 各层职责

| 层 | 类型 | 职责 | 刻意不知道 |
|----|------|------|------------|
| 持久化 | `LocalDbContext` | Note/Folder 实体 | UI 选中态 |
| 同步 | `SyncService` | Push/Pull、`_syncLock`、写 DB 后 `Publish` | FolderTree、Dispatcher |
| 读模型 | `INavigationReadModel` | 发布「导航投影应如何变」 | 如何 patch 树节点 |
| 本地变更 | `INavigationMutationPublisher` | CRUD/过滤/布局的统一出口 | Sync 协议 |
| 协调 | `NavigationPresentationCoordinator` | deferred、三分投影分发 | REST/Hub |
| 导航 Store | `INavigationStore` / `NavigationStore` | 选中解析、增量/fallback Refresh | 同步何时完成 |
| 编辑器会话 | `IEditorSessionService` | 版本感知 reload | Pull 细节 |
| VM | `FolderTree` / `NoteList` | 树/列表 UI 状态 | 不应再 scattered 调用 `Refresh()` |

## 三、为什么分 P1 → P3 → P4 → P5

演进不是重写，而是**逐层还债**：每一阶段解决一类结构性问题，并保留上一阶段成果作为 fallback。

### P1：编排与合并规则（2025）

**问题**：`MainViewModel` 里散落 Refresh；`NoteList` 与 `Editor` 各自合并 DB 字段，规则不一致；树模式选中引用错误。

**设计**：

- `INavigationStore` 集中「同步后刷新 + 按 Id 选中」。
- `LocalNoteMerger` 作为 **DB → 内存** 唯一合并规则（含「DB 空不覆盖非空正文」）。
- `SelectNoteById` 强制从 `NoteList.Notes` 解析，避免树节点引用污染列表。

**为何这样**：先把**不变量**（选中引用、合并语义）钉死，否则后续增量 patch 会在错误对象上合并。

---

### P3：增量 patch 替代全量 Refresh（2026）

**问题**：P1 仍每次同步 `FolderTree.Refresh()` + `NoteList.Refresh()`，SignalR 每条变更触发一次全量重建。

**设计**：

- `SyncNavigationDelta` / `NavigationChange` 描述「本批变更对导航的影响」。
- `NavigationPatcher` + `ApplyNavigationDelta` 对树/列表做**局部 patch**。
- `NavigationRefreshPolicy` 在搜索、重命名、文件夹 move、大批量时 **fallback 全量**（正确性优先）。

**为何这样**：全量 Refresh 实现简单但 O(整树)；小批量远端变更占绝大多数，增量 patch 成本低。fallback 保证复杂场景不必猜 UI 状态。

**仍存在的债**：Sync 仍用 `OnSyncComplete` 事件直驱 UI；编辑器 reload 混在 `NavigationPatcher` 里。

---

### P4：读模型隔离 Sync 与展示（2026）

**问题**：`_syncLock`、`IsEditingSessionActive` 是**事后补丁**——Sync 完成与 UI 刷新仍同一条事件总线，编辑器清空/光标跳动是**耦合症状**而非单点 bug。

**设计**：

```
Sync 写 DB → INavigationReadModel.Publish → Coordinator → Store + EditorSession
```

- 删除 `SyncService.OnSyncComplete`；`MainViewModel` 不再订阅同步完成。
- `NavigationPresentationCoordinator` 独占 **Dispatcher、2s deferred、编辑保护**。
- `IEditorSessionService` 跟踪 `loadedVersion`，**版本未前进不 reload**。
- `NavigationPatcher` 不再碰 `EditorViewModel`。

**为何这样**：

| 决策 | 理由 |
|------|------|
| 读模型边界 | Sync 只表达「DB 变了什么」，不表达「怎么刷 TreeView」 |
| Coordinator 单入口 | deferred/合并 delta 只写一处，避免 Store 与 MainVM 重复 |
| 编辑器会话层 | 编辑器是长生命周期会话，与「导航列表投影」生命周期不同 |
| 版本门控 | 避免 metadata 抖动触发 `ContentRestored` 重置 AvalonEdit 光标 |

---

### P5：单向数据流 + 三分投影（2026）

**问题**：P4 只隔离了**远端 Sync**；本地新建文件夹、搜索框、布局切换仍直接 `Refresh()`，破坏「唯一入口」。

**设计**：

```csharp
enum NavigationProjectionKind
{
    DataChange,     // 远端 + 本地 CRUD → ApplyRemoteDelta + EditorSession
    FilterOnly,     // 搜索 → 只 ApplyFilterProjection，不碰编辑器
    LayoutRebuild   // 模式切换/回收站 → 全量重建导航，仍不经 Sync 事件
}
```

- `INavigationMutationPublisher`：FolderTree / NoteList 本地 CRUD **唯一出口**。
- `NavigationFilterBridge`：`SearchChanged` → `NotifyFilterChanged()`。
- 本地与远端 **共用** `NavigationPatcher` 增量路径。

**为何这样**：

| 决策 | 理由 |
|------|------|
| 三分投影 | 搜索/filter 不应触发编辑器 reload；布局切换需要全量重建但不应假装是 Sync |
| 本地也走读模型 | 否则「远端增量、本地全量」双轨，行为不一致且难测 |
| `waitForPresentation: true`（本地） | CRUD 后 VM 需立即 `FindNodeById` 进入重命名，必须同步投递到 UI 线程 |
| 保留 VM 内 `Refresh()` 作实现细节 | fallback 与 Filter/Layout 仍复用同一重建逻辑，避免三份复制 |

## 四、现行数据流（端到端）

### 4.1 远端同步

```
SignalR / Pull / Push
  → SyncService 写 SQLite（_syncLock）
  → NavigationDeltaBuilder（Apply 前检测 ParentFolderChanged）
  → INavigationReadModel.Publish(DataChange)
  → Coordinator
       ├─ IsEditingSessionActive / 内联重命名 → 2s deferred + merge delta
       ├─ NavigationStore.ApplyRemoteDelta
       │     ├─ NavigationPatcher 增量成功
       │     └─ 否则 fallback Refresh(All)
       └─ EditorSession.ApplyRemoteChanges（版本门控 + IsEditingSessionActive）
```

### 4.2 本地 CRUD（用户新建/删除/重命名/收藏）

```
FolderTreeViewModel / NoteListViewModel
  → 写 SQLite + LocalChangeTracker
  → INavigationMutationPublisher.Notify*
  → ReadModel.Publish(DataChange, waitForPresentation: true)
  → Coordinator（同 4.1 数据路径，不经 SyncService）
```

### 4.3 搜索

```
NavigationViewModel.SearchText 变化
  → NavigationFilterBridge
  → NotifyFilterChanged() → FilterOnly
  → NavigationStore.ApplyFilterProjection（Refresh 树/列表，不碰 Editor）
```

### 4.4 布局 / 回收站 / 冲突对话框

```
NotifyLayoutRebuild() → LayoutRebuild
  → NavigationStore.ApplyLayoutRebuild
```

## 六、编辑保护（为何仍需要 IsEditingSessionActive）

读模型隔离后，**仍需要**编辑会话判断，原因不是架构失败，而是 **AvalonEdit 语义**：

- 给 `TextEditor.Text` 赋值会重置 caret/scroll，即使字符串相同。
- 用户停笔后 `IsDirty` 可能已为 false，但焦点仍在编辑器内。

因此 Coordinator 在 `DataChange` 路径上：

1. `IsEditingSessionActive` → deferred，合并 pending delta。  
2. `TryReloadCurrentNoteFromDb` 仅在 `EditorContent == DB` 一致时跳过 `ContentRestored`。  
3. `EditorSession` 用 `loadedVersion` 跳过无实质变化的 reload。

这是 **展示层策略**，与 Sync 层 `_syncLock`（DB 并发）正交。

---

## 七、Fallback 策略（何时仍全量 Refresh）

增量 patch 是**优化路径**，不是唯一路径。满足任一条件仍 `Refresh(All)`：

| 条件 | 原因 |
|------|------|
| `RequiresFullRefresh` | 全量同步、冲突 Keep Remote 等 |
| 搜索非空 | 过滤结果无法从单条 delta 推断 |
| 内联重命名 | 避免打断 IME |
| `ParentFolderChanged` | 树需重挂 |
| 变更数 > 50 | patch 复杂度超过重建 |
| 树模式笔记跨文件夹 move | 树内节点需 fallback |

**设计原则**：宁可 fallback 全量，不可 patch 错节点导致选中/编辑器状态损坏。
