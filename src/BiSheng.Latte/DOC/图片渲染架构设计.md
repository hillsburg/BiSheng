# 图片渲染架构设计

## 概述

BiSheng 编辑器的图片系统分为三个独立层次：**渲染层**（BiSheng.Editor）、**桥接层**（BiSheng.Latte）、**同步层**（BiSheng.Server）。各层通过接口和事件松耦合，编辑器程序集不依赖任何宿主实现。

## 架构分层

```
BiSheng.Editor（渲染层）
├── ImageRenderer          ← VisualLineElementGenerator，解析 ![alt](url) 并创建 InlineObjectElement
├── IImageResolver         ← 接口：宿主实现，解析自定义 URI
└── ImageResolveStatus     ← 枚举：Ready / Loading / Unavailable

BiSheng.Latte（桥接层）
├── LatteImageResolver     ← IImageResolver 实现，桥接编辑器与同步服务
├── ImageStorageService    ← 本地 SQLite 元数据管理
└── ImageSyncService       ← 上传/下载/增量拉取

BiSheng.Server（服务端）
└── ImagesController       ← 图片上传/下载 API
```

## 核心组件

### ImageRenderer（图片渲染器）

**位置：** `BiSheng.Editor/Rendering/ImageRenderer.cs`

继承 AvalonEdit 的 `VisualLineElementGenerator`，在 VisualLine 构建阶段拦截 `![alt](url)` 语法，创建 `InlineObjectElement`（内联 WPF 控件）。

**职责：**
- 解析 Markdown 图片语法 `![alt](url)`
- 根据 URL 类型分发处理（bisheng:// / http(s):// / 本地路径）
- 管理图片缓存（`Dictionary<string, BitmapSource>`）
- 创建图片 UI 或占位符 UI

**URL 分发逻辑：**
```
![alt](url)
  ├── http:// / https:// → LoadNetworkImageAsync → 缓存 → 重绘
  ├── bisheng://img/{uuid} → IImageResolver.ResolveLocalPath → 加载 / 占位符
  ├── 本地路径 → LoadLocalImage → 缓存
  └── 其他 → "图片不存在" 占位符
```

### IImageResolver（图片解析器接口）

**位置：** `BiSheng.Editor/Rendering/IImageResolver.cs`

编辑器定义的纯接口，不依赖任何宿主实现。由 BiSheng.Latte 的 `LatteImageResolver` 实现并注入。

```csharp
public interface IImageResolver
{
    string? ResolveLocalPath(string uri);   // URI → 本地路径（null = 未就绪）
    ImageResolveStatus GetStatus(string uri);  // Ready / Loading / Unavailable
    void RequestDownload(string uri);          // 触发后台下载
}
```

### LatteImageResolver（桥接实现）

**位置：** `BiSheng.Latte/Services/LatteImageResolver.cs`

桥接 ImageRenderer 和 ImageSyncService，将 `bisheng://img/{uuid}` 解析为 `images/{uuid}.png` 路径。下载完成后通过 `OnImageResolved` 事件通知 UI 重绘。

### ImageSyncService（图片同步服务）

**位置：** `BiSheng.Latte/Services/ImageSyncService.cs`

独立于笔记文本同步，专门处理图片的上传和下载。

| 功能 | 触发方式 | 频率 |
|------|---------|------|
| 上传 | 定时器扫描 `LocalImage` 表中 `Synced=false` 的记录 | 每 10 秒 |
| 按需下载 | `LatteImageResolver.RequestDownload` 调用 | 实时 |
| 增量拉取 | 定时器检查服务端新增图片 | 每 60 秒 |

## 图片粘贴流程

```
用户 Ctrl+V
    │
    ▼
MarkdownEditorControl.HandlePasteImage()
    ├── ClipboardHelper.HasImageInClipboard()
    ├── 生成 UUID，保存到 images/{uuid}.png
    ├── >10MB 自动压缩（ClipboardHelper.CompressImage）
    ├── 插入文本: ![image](bisheng://img/{uuid})
    └── 触发事件: ImagePasted?.Invoke(imageId, filePath)
            │
            ▼
    MainWindow（宿主）
        └── ImageStorageService.RecordImage(imageId, noteId, filePath)
                └── 写入 LocalImage 表，Synced=false
```

## 图片显示尺寸策略

`ImageRenderer.CreateImageBorder` 中动态计算显示尺寸：

```csharp
double availableWidth = editor.ActualWidth - 40;  // 编辑器可用宽度
double imgWidth = bitmap.PixelWidth;               // 图片原始宽度

// 小图原样展示，大图等比缩放
double displayMaxWidth = imgWidth <= availableWidth ? imgWidth : availableWidth;
```

- **小图**（`PixelWidth ≤ availableWidth`）：`MaxWidth = PixelWidth`，原始尺寸
- **大图**（`PixelWidth > availableWidth`）：`MaxWidth = availableWidth`，等比缩放
- 高度由 `Stretch.Uniform` 自动计算，无 `MaxHeight` 限制
- 解码时不设 `DecodePixelWidth`，保留原始分辨率以供尺寸判断

## 行间距与图片定位

### 问题背景

AvalonEdit 的 `InlineObjectElement` 将图片作为内联 UI 元素嵌入 TextLine。当启用自定义 `LineSpacing` 时，图片行的槽位高度计算需要与普通文本行保持一致的间距策略。

### 固定间隙法

每行 TextLine 的槽位 = `textLine.Height + gap`，其中 `gap` 是固定值：

```
gap = defaultLineHeight × (lineSpacing - 1.0)
```

无论 TextLine 包含文字还是图片，间隙大小恒定。避免了乘法方案（`height × lineSpacing`）导致图片行间隙过大的问题。

**对比：**

| 方案 | 普通文本行（20px） | 图片行（400px） | 间隙差异 |
|------|-------------------|----------------|---------|
| 乘法 `height × 1.5` | 槽位 30px，间隙 10px | 槽位 600px，间隙 **200px** | ✗ 不一致 |
| 固定间隙 `height + 10` | 槽位 30px，间隙 10px | 槽位 410px，间隙 **10px** | ✓ 一致 |

### 需要保持一致的位置

以下所有位置统一使用 `height + gap`：

1. `VisualLine.SetTextLines` — 计算 VisualLine 总高度
2. `VisualLine.GetTextLineVisualYPosition` — Y 坐标定位
3. `VisualLine.GetTextLineByVisualYPosition` — 点击命中测试
4. `VisualLineDrawingVisual` — 文字绘制偏移
5. `TextView.ArrangeOverride` — InlineObjectElement 定位

## 图片缓存机制

```
ImageRenderer._imageCache: Dictionary<string, BitmapSource>
```

- **本地图片**：加载后立即缓存，后续直接命中
- **网络图片**：首次显示占位符，异步下载完成后写入缓存并触发 `Redraw()`
- **bisheng:// 图片**：解析为本地路径后加载并缓存
- 缓存无过期策略，图片在编辑器生命周期内常驻内存

## 设计决策

### 为什么使用 `bisheng://` 自定义 URI 而非直接存路径？

| 考量 | 说明 |
|------|------|
| 跨设备同步 | 本地路径在不同设备上不同，UUID 全局唯一 |
| 解析延迟 | 图片可能尚未下载到本地，需要异步解析 |
| 状态管理 | 通过 `ImageResolveStatus` 统一表达 Ready/Loading/Unavailable |

### 为什么不实现行内编辑模式？

| 问题 | 说明 |
|------|------|
| 坐标体系复杂 | 编辑模式下图片消失→行高变化→所有后续行的 InlineObjectElement 需要重新布局 |
| 缓存失效 | 行号变化时需清除缓存重新构建 VisualLine，影响性能 |
| 使用频率低 | 用户修改图片 URL 的场景极少，删除后重新粘贴更直观 |

### 为什么图片同步独立于笔记文本同步？

| 考量 | 说明 |
|------|------|
| 粒度不同 | 文本是增量 diff 同步，图片是整张上传/下载 |
| 频率不同 | 文本频繁变更，图片偶尔新增 |
| 带宽管理 | 图片较大，需独立控制上传节奏，避免阻塞文本同步 |
| 失败隔离 | 图片上传失败不影响笔记文本同步 |
